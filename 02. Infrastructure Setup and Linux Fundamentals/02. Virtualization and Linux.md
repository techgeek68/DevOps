## Theory

## 1. Virtualization

Virtualization is the process of creating software based versions of physical resources such as servers, storage, and networks. It allows multiple operating systems and applications to run on a single physical machine, making better use of hardware that would otherwise sit idle.

Before virtualization, a physical server typically ran one operating system and one workload. The hardware was underutilized, expensive to maintain, and difficult to reproduce. Virtualization solved all three problems at once.

---

### Why Virtualization Matters for DevOps?

| Benefit | Practical Meaning |
|---|---|
| Resource efficiency | Run multiple VMs on one physical machine instead of one workload per server |
| Isolation | A fault, crash, or misconfiguration in one VM does not affect others |
| Environment consistency | Recreate identical dev, test, staging, and production environments from the same base image |
| Faster provisioning | Deploy a pre configured VM in minutes instead of hours of manual setup |
| Scalability | Spin VMs up or down to match workload demand without touching hardware |
| Cost reduction | Fewer physical servers, lower power consumption, less data center space |
| Disaster recovery | VM snapshots capture state at a point in time; roll back to a clean state instantly if something breaks |

In a DevOps context, virtualization is the foundation for lab environments, CI/CD build agents, Kubernetes nodes, and local development machines that mirror production.

---

## 2. Hypervisors

A hypervisor (also called a Virtual Machine Monitor) is the software layer that creates and manages virtual machines. It sits between the physical hardware and the guest operating systems, allocating CPU, RAM, storage, and network resources to each VM and keeping them isolated from one another.

### Type 1: Bare Metal Hypervisor

A Type 1 hypervisor runs directly on the physical hardware. There is no underlying host operating system. The hypervisor itself is the operating system, and VMs run on top of it.

Type 1 hypervisors offer higher performance, stronger isolation, and better security because no host OS layer can be compromised or consume resources.

Type 1 hypervisors are used in production data centers, cloud provider infrastructure, and enterprise server virtualization.

Examples: VMware ESXi, Microsoft Hyper-V, Xen, KVM (Kernel-based Virtual Machine), Proxmox

**KVM note:** KVM is a Linux kernel module, integrated into the kernel since version 2.6.20. When KVM is loaded, the Linux kernel itself becomes a Type 1 hypervisor with direct access to hardware virtualization extensions (Intel VT-x, AMD-V). QEMU is a separate user space tool that works alongside KVM to handle device emulation. The combination is referred to as KVM/QEMU. KVM alone is the hypervisor; QEMU provides the hardware interface that makes it usable.

**Hyper V note:** Microsoft Hyper V is a Type 1 hypervisor. When Hyper V is enabled on a Windows host, the hypervisor inserts itself between the hardware and the OS. The Windows installation that you see becomes a privileged management partition, not a standard host OS. This is confirmed in the official Microsoft documentation: "Hyper V features a Type 1 hypervisor based architecture."

---

### Type 2: Hosted Hypervisor

A Type 2 hypervisor runs as an application on top of an existing host operating system. The host OS manages hardware resources, and the hypervisor sits on top of it.

Type 2 hypervisors are easier to install and use, making them well-suited for local development, student lab environments, and desktop virtualization. They add overhead compared to Type 1 because all hardware access passes through the host OS.

Examples: VMware Workstation Pro, VirtualBox, Parallels Desktop, UTM

### Comparison

| Feature | Type 1 (Bare Metal) | Type 2 (Hosted) |
|---|---|---|
| Runs on | Physical hardware directly | Host operating system |
| Performance | High; minimal overhead | Moderate; host OS adds overhead |
| Security | Stronger isolation | Dependent on host OS security |
| Setup complexity | Higher; dedicated hardware needed | Low; installs like any application |
| Use case | Production servers, cloud infrastructure | Developer labs, learning environments |
| Examples | VMware ESXi, Hyper-V, KVM | VMware Workstation Pro, VirtualBox |

---

### VMs vs. Containers

Both VMs and containers provide isolation, but they work differently and serve different purposes.

| Aspect | Virtual Machine | Container |
|---|---|---|
| Isolation unit | Full OS + kernel | Process namespace on shared host kernel |
| Size | Gigabytes | Megabytes |
| Startup time | Minutes | Seconds |
| OS | Each VM has its own OS | All containers share the host kernel |
| Security isolation | Strong hardware-level isolation | Process-level isolation |
| Use case | Full system environments, multiple OS types | Microservices, CI/CD, packaging applications |

VMs and containers are complementary. In this course, you run containers (Module 5) inside VMs (this module). In production, many organizations run containers inside VMs on cloud infrastructure.

---

## WSL vs VM

| Feature | WSL2 | VM |
|---|---|---|
| **Type** | Lightweight Hyper V VM with real Linux kernel | Full hardware emulation |
| **Kernel** | Microsoft maintained Linux kernel fork | Full guest OS kernel |
| **Performance** | Near native | Hypervisor overhead |
| **Boot time** | Seconds | Minutes |
| **Resource usage** | Lightweight | Heavy (dedicated RAM/CPU) |
| **GUI support** | Good (via WSLg) | Full desktop support |
| **Isolation** | Partial (Hyper-V boundary) | Full |
| **Networking** | vEth + NAT (mirrored mode available) | Separate network stack |
| **File system** | Own ext4 disk + `/mnt/c` Windows access | Separate disk image |
| **Snapshot/rollback** | N/A | Available |
| **Hardware access** | Limited | Depends on hypervisor |
| **Use case** | Dev tools, CLI, scripting | Full OS testing, labs, servers |

> WSL2 is a real Linux kernel in a lightweight VM, optimized for developer workflows. VMs offer full isolation and hardware control, better suited for infrastructure and lab environments.

# Lab 2.2.A: VM Setup and OS Installation

## Objective

Install VMware Workstation Pro, create a virtual machine, install CentOS Stream 10 (or CentOS Stream 9), and configure the system for use as a DevOps lab server.

This VM becomes the base image from which all other lab VMs in this course are cloned.

---

## Part 1: Install VMware Workstation Pro

1. Go to **support.broadcom.com** and create a free Broadcom account if you do not already have one.
2. Search for **VMware Workstation Pro** under the software downloads section.
3. Download the installer for your host OS (Windows or Linux).
4. Run the installer and follow the prompts. When asked for a license key, select the **Personal Use** option. No key is required.
5. Reboot your host machine if prompted.

---

## Part 2: Download CentOS Stream 10 ISO

1. Go to **centos.org** and navigate to the CentOS Stream 10 download page.
2. Download the **DVD ISO** (full installation image, approximately 10 GB).
3. Save the ISO to a location you can find easily (for example: `D:\ISOs\CentOS-Stream-10-x86_64-dvd1.iso`).

CentOS Stream 10 was released on December 12, 2024, and is maintained until 2030. It is the current upstream development branch for RHEL 10. CentOS Stream 9 is also acceptable and remains supported until May 2027.

---

## Part 3: Create the Virtual Machine

**Step 1: Launch the New VM Wizard**

Open VMware Workstation Pro. Click **File > New Virtual Machine**. Select **Typical (recommended)** and click **Next**.

**Step 2: Set the Installation Source**

Select **I will install the operating system later**. Click **Next**.

Setting the ISO now causes VMware to attempt "Easy Install," which automates the installation in a way that skips important configuration steps. Attach the ISO manually after the VM is created.

**Step 3: Specify the Guest OS Type**

- Guest operating system: **Linux**
- Version: **Red Hat Enterprise Linux 9 64-bit** (the closest option to CentOS Stream 10)

Click **Next**.

**Step 4: Name the VM and Choose a Location**

- Name: `DevOps-BaseVM` (or another descriptive name)
- Location: Choose a folder with enough free space. Create a `DevOps-Lab` folder on your largest available drive.

Click **Next**.

**Step 5: Set Disk Capacity**

- Disk size: **40 GB minimum** (60 GB recommended if disk space allows)
- Select **Store virtual disk as a single file**

Click **Next**, then **Finish**.

**Step 6: Edit Hardware Before First Boot**

Before powering on, click **Edit virtual machine settings** and configure the following:

| Component | Recommended Setting |
|---|---|
| Memory (RAM) | 4 GB (4096 MB) minimum. 8 GB recommended if your host has 16 GB or more |
| Processors | 2 cores |
| CD/DVD (IDE or SATA) | Set to **Use ISO image file**; browse to your CentOS Stream 10 ISO |
| Network Adapter | NAT (provides internet access through the host; simplest for lab use) |

Click **OK**.

---

## Part 4: Install CentOS Stream 10

**Step 1: Power On and Boot**

Power on the VM. The VM will boot from the ISO. At the boot menu, select **Install CentOS Stream 10** and press Enter.

**Step 2: Language and Keyboard**

Select your language (English) and keyboard layout. Click **Continue**.

**Step 3: Installation Summary Screen**

This screen shows all configuration categories. You must configure the ones marked with a warning icon before installation can begin.

**Installation Destination**

Click **Installation Destination**. Select the 40 GB virtual disk. Under Storage Configuration, leave **Automatic** selected. VMware Workstation creates an optimized virtual disk that does not require manual partitioning for this course.

Click **Done**.

The automatic partitioning layout will be:

| Partition | Size |
|---|---|
| /boot | 1 GB |
| swap | 2 GB |
| / (root) | Remaining space |

**Software Selection**

Click **Software Selection**. Select **Server with GUI** as the base environment. This installs a full graphical desktop and the server tools you need for lab work.

Click **Done**.

**Time and Date**

Click **Time and Date**. Set the region to **Asia** and the city to **Kathmandu**. Enable network time (NTP) if the toggle is available.

Click **Done**.

**Network and Host Name**

Click **Network and Host Name**. Toggle the network interface to **ON**. Set the hostname to `basevm.local` or simply `localhost`.

Click **Done**.

**Step 4: Root Password and User Creation**

Set a root password. Make it something you will remember for lab use.

Create a regular user account. Check **Make this user administrator** so the account has sudo access.

**Step 5: Begin Installation**

Click **Begin Installation**. The installation takes approximately 10 to 20 minutes depending on your hardware. The VM will reboot when complete.

---

## Part 5: Post-Installation Configuration

Log in with the user account you created.

**Update all packages:**

```bash
sudo dnf update -y
```

This may take several minutes on first run. Run it after every fresh installation before doing any further configuration.

**Enable and start SSH:**

```bash
sudo systemctl enable sshd
sudo systemctl start sshd
sudo systemctl status sshd
```

SSH should already be running, but confirm it. You will use SSH to connect to cloned VMs from your host terminal.

**Allow SSH through the firewall:**

```bash
sudo firewall-cmd --permanent --add-service=ssh
sudo firewall-cmd --reload
sudo firewall-cmd --list-all
```

**Install VMware Tools (open-vm-tools):**

```bash
sudo dnf install -y open-vm-tools
sudo systemctl enable vmtoolsd
sudo systemctl start vmtoolsd
```

VMware Tools improves VM performance, enables copy-paste between host and guest, and allows shared folders if you need them.

**Verify the installation:**

```bash
# Check OS version
cat /etc/os-release

# Check IP address (note it for SSH access)
ip addr show

# Check disk usage
df -h

# Check memory
free -h
```

Record the VM's IP address. You will SSH into cloned VMs using the management network in later labs.

**Take a snapshot before doing anything else:**

In VMware Workstation, go to **VM > Snapshot > Take Snapshot**. Name it `Clean Install`. This is your fallback point. If anything goes wrong in any subsequent lab, you can revert to this snapshot and start fresh.

# Lab 2.2.B: Clone VMs for the Lab Environment

## Objective

Clone the base VM to create all the individual server VMs needed for this course. Cloning is faster and more reliable than installing each VM from scratch.

## Prerequisites

The base VM (`DevOps-BaseVM`) must be:
- Fully installed and updated
- Powered off (not suspended, fully powered off)
- In a clean state (nothing course-specific configured yet)

Always clone from a clean base. If you have already configured something on the base VM, revert to the `Clean Install` snapshot before cloning.

---

The following VMs are needed for this course. All are cloned from `DevOps-BaseVM`.

| VM Name | Role | RAM | vCPUs | Disk |
|---|---|---|---|---|
| control-node | Developer / Ansible control node | 2 GB | 2 | 40 GB |
| web-server | Apache / Nginx web server | 2 GB | 2 | 40 GB |

## How to Clone a VM

**Step 1: Power off the base VM**

Right-click `DevOps-BaseVM` in the VM library. Select **Power > Shut Down Guest**. Wait for it to fully power off.

**Step 2: Open the Clone Wizard**

Right-click `DevOps-BaseVM`. Select **Manage > Clone**.

**Step 3: Clone From**

Select **The current state of the virtual machine** (or select the `Clean Install` snapshot if you took one). Click **Next**.

**Step 4: Clone Type**

Select **Full Clone**. A full clone is a completely independent copy of the VM. Changes to the clone do not affect the base VM, and vice versa. Full clones are recommended for lab VMs where you need complete independence.

Click **Next**.

**Step 5: Name and Location**

Enter the VM name (for example, `jenkins-controller`) and choose its storage location under your `DevOps-Lab` folder. Click **Finish**.

The clone process takes 1 to 5 minutes depending on disk speed.

## Post-Clone Configuration

Every cloned VM needs unique identification. Without this, VMs on the same network will conflict with each other.

**Step 1: Power on the cloned VM**

**Step 2: Change the hostname**

```bash
sudo hostnamectl set-hostname jenkins-controller
```

Replace `jenkins-controller` with the appropriate name for each VM. Verify the change:

```bash
hostnamectl
```

The new hostname takes effect immediately for new shell sessions. Log out and back in to see it in the prompt.

**Step 3: Verify or change the IP address**

By default, VMware assigns a new IP via DHCP for each clone. Check the current IP:

```bash
ip addr show
```

For most labs, a DHCP-assigned IP is acceptable. For VMs that other machines will connect to repeatedly (Jenkins controller, Kubernetes control plane), assign a static IP to prevent it changing after a reboot.

To assign a static IP using NetworkManager:

```bash
# List connections
nmcli connection show

# Set a static IP on the connection (replace values to match your network)
sudo nmcli connection modify "Wired connection 1" \
  ipv4.method manual \
  ipv4.addresses 192.168.150.10/24 \
  ipv4.gateway 192.168.150.2 \
  ipv4.dns "8.8.8.8,8.8.4.4"

# Restart the connection to apply
sudo nmcli connection down "Wired connection 1"
sudo nmcli connection up "Wired connection 1"

# Confirm
ip addr show
```

The VMware NAT default gateway is usually `192.168.x.2`. Check yours in **VMware Workstation > Edit > Virtual Network Editor**.

**Step 4: Generate a new machine ID (if needed)**

In most cases, VMware generates a new MAC address for each clone automatically. Verify this in **VM Settings > Network Adapter > Advanced > MAC Address**. If the MAC address matches the base VM, click **Generate** to create a new one.

If you need to regenerate the machine ID (required for some Kubernetes configurations):

```bash
sudo rm /etc/machine-id
sudo systemd-machine-id-setup
cat /etc/machine-id
```

## Taking Snapshots

Before making significant changes to any VM (installing software, changing configuration, running a new lab), take a snapshot.

```
In VMware Workstation: VM > Snapshot > Take Snapshot
Name the snapshot descriptively: "Before Jenkins Install" or "Module 3 Start"
```

To revert to a snapshot:

```
VM > Snapshot > Snapshot Manager
Select the snapshot > Click "Go To"
```

Snapshots are not a substitute for backups, but in a lab environment they are the fastest way to recover from a mistake.

## Managing VMs

**Delete a VM permanently:**

Right-click the VM in the library. Select **Power > Shut Down Guest** if it is running. Then right-click and select **Manage > Delete from Disk**. Confirm by clicking **Yes**.

This permanently removes all VM files including snapshots. It cannot be undone.

**Remove from library without deleting files:**

Right-click the VM and select **Remove** (not "Delete from Disk"). The VM files remain on disk and can be re-added later via **File > Open**.

**Pause vs. Power Off:**

- **Suspend** saves the current memory state to disk and pauses the VM. Resume is fast but consumes disk space.
- **Power Off** cleanly shuts down the guest OS. Takes longer to start again but uses no disk space for the saved state.

For lab VMs you are not actively using, power them off completely to free RAM on your host.
