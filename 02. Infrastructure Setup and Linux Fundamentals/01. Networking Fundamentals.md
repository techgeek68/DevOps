## 1. OSI Model

The OSI (Open Systems Interconnection) model is a conceptual framework that describes how data travels from one machine to another across a network. It breaks the communication process into seven layers, each with a specific responsibility. Each layer serves the one above it and depends on the one below it.

OSI was developed by the International Organization for Standardization (ISO) and first published as ISO 7498 in 1984, then revised and reissued as ISO/IEC 7498-1 in 1994. It is a reference model, not an implementation. No real system implements OSI directly. Its value is as a diagnostic and design framework.

### The Seven Layers

| Layer | Name         | What It Does                                                   | Examples                             |
|-------|--------------|----------------------------------------------------------------|--------------------------------------|
| 7     | Application  | Interface between the network and the application              | HTTP, HTTPS, SSH, DNS, FTP, SMTP     |
| 6     | Presentation | Data encoding, encryption, and compression                     | TLS/SSL, JPEG, JSON serialization    |
| 5     | Session      | Manages communication sessions between applications            | NetBIOS, RPC                         |
| 4     | Transport    | End to end delivery, error checking, and flow control          | TCP, UDP                             |
| 3     | Network      | Logical addressing and routing between networks                | IP, ICMP, OSPF, BGP                  |
| 2     | Data Link    | Physical addressing and reliable transfer across a single link | Ethernet, MAC addresses, ARP, VLANs  |
| 1     | Physical     | Transmits raw bits over a physical medium                      | Cables (Cat6, fiber), Wi-Fi signals  |

---

### Why OSI Matters in DevOps?

When something stops working in a networked system, the OSI model provides a structured starting point. You test from the bottom up instead of guessing:

- Layer 1/2 problem: Is the cable connected? Is the NIC up? (`ip link show`)
- Layer 3 problem: Can the host reach its gateway? (`ping 192.168.1.1`, `ip route`)
- Layer 4 problem: Is the port open and the service listening? (`ss -tlnp`, `telnet <host> <port>`)
- Layer 7 problem: Is the application responding correctly? (`curl -v`, browser developer tools)

This method isolates failures to one layer at a time and eliminates guesswork.

---

### Where Major DevOps Tools Operate?

| Tool or Concept                                   | OSI Layer       |
|---------------------------------------------------|-----------------|
| Ethernet switch                                   | Layer 2         |
| Router, IP routing, AWS VPC routing               | Layer 3         |
| TCP connections, L4 load balancer                 | Layer 4         |
| Nginx, Apache, Jenkins, Kubernetes Ingress (L7)   | Layer 7         |
| TLS/SSL certificates                              | Layer 6/7       |
| Firewall rules (AWS Security Groups, firewalld)   | Layers 3 and 4  |

## 2. TCP/IP Fundamentals

### The TCP/IP Stack

TCP/IP is the protocol suite that runs the Internet and most modern networks. Unlike OSI, it is an implementation. It has four layers that map roughly onto the seven OSI layers:

| TCP/IP Layer   | Equivalent OSI Layers | Key Protocols                      |
|----------------|-----------------------|------------------------------------|
| Application    | 5, 6, 7               | HTTP, HTTPS, SSH, DNS, SMTP, FTP   |
| Transport      | 4                     | TCP, UDP                           |
| Internet       | 3                     | IP (IPv4, IPv6), ICMP              |
| Network Access | 1, 2                  | Ethernet, ARP, Wi-Fi               |

---

### TCP vs. UDP

These are the two protocols at the Transport layer. Understanding the difference is essential for reading firewall rules, diagnosing connection failures, and configuring services.

| Feature     | TCP (Transmission Control Protocol)               | UDP (User Datagram Protocol)                         |
|-------------|---------------------------------------------------|------------------------------------------------------|
| Connection  | Connection oriented (three way handshake)         | Connectionless                                       |
| Reliability | Guaranteed delivery, retransmission on loss       | No guarantee; packets may be lost or arrive out of order |
| Order       | Packets arrive in sequence                        | No ordering guarantee                                |
| Speed       | Slower due to overhead                            | Faster; minimal overhead                             |
| Use cases   | HTTP/HTTPS, SSH, database connections             | DNS queries, video streaming, VoIP, metrics over UDP |

The TCP three-way handshake: SYN > SYN-ACK > ACK. This is what happens every time your browser connects to a web server or your Jenkins agent connects to the controller. When a connection hangs, checking whether the handshake completes is the first diagnostic step.

---

### IPv4 Addressing

An IPv4 address is a 32-bit number written in four octets of decimal, separated by dots: `192.168.1.10`.

Each octet represents 8 bits, giving a range of 0 to 255 per octet. Total IPv4 address space: 2^32 = approximately 4.3 billion addresses.

An IP address has two parts: the network portion (which network) and the host portion (which device on that network). The subnet mask separates them.

---

### Subnetting and CIDR Notation

CIDR (Classless Inter Domain Routing) notation expresses both the IP address and the subnet mask in one compact form: `192.168.1.0/24`.

The `/24` is the prefix length: the number of bits used for the network portion. The remaining bits are for hosts.

| Prefix | Subnet Mask       | Network Bits | Host Bits | Usable Hosts |
|--------|-------------------|--------------|-----------|--------------|
| /8     | 255.0.0.0         | 8            | 24        | 16,777,214   |
| /16    | 255.255.0.0       | 16           | 16        | 65,534       |
| /24    | 255.255.255.0     | 24           | 8         | 254          |
| /26    | 255.255.255.192   | 26           | 6         | 62           |
| /30    | 255.255.255.252   | 30           | 2         | 2            |
| /32    | 255.255.255.255   | 32           | 0         | 1 (single host) |

Usable hosts = 2^(host bits) - 2. You subtract 2 because the first address is the network address and the last is the broadcast address. Neither can be assigned to a device.

---

**Worked example: 192.168.1.0/26**

- Prefix length: 26 bits network, 6 bits host
- Subnet mask: 255.255.255.192 (binary: 11111111.11111111.11111111.11000000)
- Network address: 192.168.1.0
- Broadcast address: 192.168.1.63 (all host bits set to 1)
- Usable host range: 192.168.1.1 to 192.168.1.62
- Number of usable hosts: 2^6 - 2 = 62

This exact calculation is Lab 2.1.B.

---

### Common Protocols and Their Ports

| Protocol       | Port | Transport   | Purpose                                                      |
|----------------|------|-------------|--------------------------------------------------------------|
| HTTP           | 80   | TCP         | Unencrypted web traffic                                      |
| HTTPS          | 443  | TCP         | TLS encrypted web traffic                                    |
| SSH            | 22   | TCP         | Encrypted remote shell access                                |
| FTP (data)     | 20   | TCP         | File transfer (data channel)                                 |
| FTP (control)  | 21   | TCP         | File transfer (control channel)                              |
| SMTP           | 25   | TCP         | Email relay between servers                                  |
| DNS            | 53   | TCP and UDP | Domain name resolution (UDP for queries; TCP for zone transfers and large responses) |
| HTTP alt       | 8080 | TCP         | Common alternative port for web apps and Jenkins             |
| HTTPS alt      | 8443 | TCP         | Common alternative port for HTTPS services                   |
| MySQL          | 3306 | TCP         | MySQL/MariaDB database                                       |
| PostgreSQL     | 5432 | TCP         | PostgreSQL database                                          |
| Redis          | 6379 | TCP         | Redis in-memory data store                                   |

Knowing these ports by memory is a practical skill. When you write a firewall rule to allow Jenkins traffic, you need to know it runs on port 8080 by default. When you debug a failed database connection, you check whether port 3306 is reachable.

---

### Routing

Routing is how a packet finds its way from source to destination across one or more networks.

A router makes forwarding decisions based on a routing table: a list of destination networks and the next hop to reach them. Each entry says: "to reach network X, send the packet to Y."

**Default gateway:** The router a host sends packets to when it has no more specific route. Every server needs a default gateway configured or it cannot reach anything outside its local subnet. On Linux: `ip route add default via <gateway-ip>`.

**Static route:** A manually configured route for a specific destination. Used in lab environments and simple networks where dynamic routing protocols are not needed.

**Dynamic routing protocols:** In large networks, routers exchange route information automatically. OSPF is common inside enterprise networks. BGP runs the internet. In DevOps work, you will mostly see static routing in lab environments and rely on cloud-managed routing in production.

**Routing in AWS VPCs:** Every VPC has a main route table. Subnets are associated with route tables. A public subnet has a route to an Internet Gateway (`0.0.0.0/0 > igw-xxxxxxxx`). A private subnet has no internet route, or routes outbound traffic through a NAT Gateway. This is exactly the setup you build in Module 6.

## 3. DNS (Domain Name System)

DNS is the system that translates human readable domain names like `github.com` into IP addresses that machines can route to. Without DNS, you would have to type `140.82.121.4` every time you wanted to visit GitHub.

Understanding DNS is important in DevOps because misconfigured or slow DNS causes service failures that are easy to misdiagnose as network or application problems.


### How DNS Resolution Works?

When you type `github.com` into a browser, this is what happens:

1. Your machine checks its local cache. If it has a recent answer, it uses that.
2. If not, it sends a query to your configured **recursive resolver** (usually your router or your ISP's DNS server, or a public resolver like `8.8.8.8` or `1.1.1.1`).
3. The recursive resolver checks its own cache. If it does not have the answer, it queries a **root name server** (there are 13 sets of these distributed globally).
4. The root server points the resolver to the **TLD (Top Level Domain) name server** for `.com`.
5. The TLD server points the resolver to the **authoritative name server** for `github.com`.
6. The authoritative name server returns the actual IP address.
7. The resolver caches the answer (for the duration of the record's TTL) and sends it back to your machine.

This entire chain typically completes in under 50 milliseconds on a healthy network.

---

### DNS Record Types

| Record | Purpose                                             | Example                              |
|--------|-----------------------------------------------------|--------------------------------------|
| A      | Maps a hostname to an IPv4 address                  | `github.com  > 140.82.121.4`         |
| AAAA   | Maps a hostname to an IPv6 address                  | `github.com  > 2606:50c0:8000::154`  |
| CNAME  | Alias from one hostname to another                  | `www.example.com  > example.com`     |
| MX     | Identifies mail servers for the domain              | `github.com  > aspmx.l.google.com`   |
| NS     | Identifies authoritative name servers for the domain | `github.com  > ns1.p16.dynect.net`  |
| TXT    | Stores arbitrary text (used for SPF, DKIM, domain verification) | `"v=spf1 include:..."`  |
| PTR    | Reverse lookup: maps an IP to a hostname            | `140.82.121.4 > github.com`         |

---

### TTL (Time to Live)

Every DNS record has a TTL value in seconds. It tells resolvers how long to cache the answer before re-querying. A TTL of `300` means the record can be cached for 5 minutes.

Why this matters in DevOps: when you change a DNS record (for example, pointing a domain to a new server after a deployment), the old answer stays cached in resolvers around the world until the TTL expires. If your TTL is 3600 (1 hour), propagation takes up to an hour. Lower the TTL to 60 or 300 before a planned migration to reduce the wait.

---

### DNS in Practice

`/etc/resolv.conf` on Linux specifies which DNS server the system uses:

* View current DNS configuration
```bash
cat /etc/resolv.conf
```

* On systems using systemd-resolved
```bash
resolvectl status
```

On RHEL/Fedora with NetworkManager, DNS is managed via `nmcli` or connection profiles, not by editing `/etc/resolv.conf` directly. The file is often a symlink managed by `systemd-resolved`.

## 4. Load Balancing

### What Is a Load Balancer?

A load balancer sits in front of a group of servers and distributes incoming requests across them. Without one, all traffic hits a single server. When that server becomes overloaded or fails, the application goes down.

With a load balancer:
- Traffic is spread across multiple servers, preventing any single one from being overwhelmed.
- If a server fails, the load balancer stops sending it traffic, and the application stays up.
- You can add or remove servers without changing anything the client sees.

### Layer 4 vs. Layer 7 Load Balancing

| Type              | Operates At    | Routing Basis                          | Awareness of Content |
|-------------------|----------------|----------------------------------------|----------------------|
| L4 (Transport)    | TCP/UDP level  | IP address and port                    | None; routes without reading the request |
| L7 (Application)  | HTTP level     | URL path, hostname, headers, cookies   | Full; can route based on request content |

An L7 load balancer can route `/api/*` requests to an API server cluster and `/static/*` to an object storage endpoint. An L4 load balancer cannot make that distinction.

Nginx functions as both. AWS Elastic Load Balancing offers an Application Load Balancer (ALB, Layer 7) and a Network Load Balancer (NLB, Layer 4).

---

### Load Balancing Algorithms

Nginx open source supports five load balancing methods: Round Robin, Least Connections, Least Time, IP Hash, and Generic Hash. The three most common ones are covered below.


**Round Robin (default)**

Requests go to servers in a rotating sequence: server 1, server 2, server 3, server 1, and so on.

Use it when servers have equal capacity and requests are roughly the same size. Round robin is the simplest and most common default. No directive is needed because it is the default behavior.

```nginx
upstream backend {
    server 192.168.1.10;
    server 192.168.1.11;
    server 192.168.1.12;
    # Round robin is the default; no directive needed
}
```

Limitation: It assumes all requests take the same time. If some requests are CPU heavy and others are trivial, round robin can overload one server while others sit idle.

---

**Least Connections**

The next request goes to the server with the fewest currently active connections.

Use it when requests vary significantly in duration (some API calls finish in 10ms, others stream data for 30 seconds). Least connections adapts automatically by routing new requests away from servers that are still busy.

```nginx
upstream backend {
    least_conn;
    server 192.168.1.10;
    server 192.168.1.11;
    server 192.168.1.12;
}
```

---

**IP Hash**

The client's source IP address is hashed to determine which server handles its requests. The same IP always maps to the same server (session persistence, or "sticky sessions").

Use it when the application keeps session state on the server, and you need the same user to always reach the same backend.

```nginx
upstream backend {
    ip_hash;
    server 192.168.1.10;
    server 192.168.1.11;
    server 192.168.1.12;
}
```

Limitation: if a server goes down, all clients mapped to it get reassigned. Also, if many users share a NAT IP (for example, an entire office behind a single public IP), one server takes all their traffic. For long-term production use, move session state to a shared store like Redis and use least connections instead.

---

**Weighted Round Robin**

Servers get a weight proportional to their capacity. A server with weight 3 receives three times as many requests as one with weight 1.

```nginx
upstream backend {
    server 192.168.1.10 weight=3;  # More powerful server
    server 192.168.1.11 weight=1;
}
```

---

### Health Checks

Any production load balancer should periodically probe each backend to confirm it is healthy and remove it from rotation if it fails.

Nginx open source uses passive health checks: it detects failures when a request to a backend actually fails. Nginx Plus adds active health checks. AWS ALB performs active health checks: it sends an HTTP request to a configured path (typically `/health`) on each backend every 30 seconds and removes backends that return non-2xx responses or time out.

## 5. Firewalls and Security Groups

### What Is a Firewall?

A firewall is a network security system that monitors and controls traffic based on a defined set of rules. Rules specify: direction (inbound or outbound), source IP, destination IP, protocol (TCP/UDP/ICMP), and port. Traffic that matches an allow rule is permitted; everything else is denied (default-deny posture) or vice versa.

### Stateless vs. Stateful Firewalls

| Type       | How It Works                                                  | Implication                                                    |
|------------|---------------------------------------------------------------|----------------------------------------------------------------|
| Stateless  | Evaluates each packet independently against rules             | You must explicitly allow both directions: inbound request AND outbound response |
| Stateful   | Tracks connection state; automatically allows return traffic  | You only allow the initiating direction; the response is automatically allowed |

AWS Security Groups are stateful. `iptables` in its basic form is stateless, but `conntrack` (connection tracking) makes it stateful. `firewalld` on RHEL/Fedora uses `nftables` with connection tracking by default, so it is stateful.

---

### `firewalld` on Linux

`firewalld` is the default firewall management tool on RHEL, Fedora, and CentOS Stream. It uses zones to classify network interfaces. The most common zones:

- `public`: the default for external-facing interfaces. Only explicitly allowed services and ports are permitted.
- `trusted`: all traffic is allowed.
- `drop`: all incoming traffic is silently dropped.

Common `firewall-cmd` operations:

* List all rules in the active zone
```bash
firewall-cmd --list-all
```

* Allow SSH (`--permanent` survives reboot)
```bash
firewall-cmd --permanent --add-service=ssh
```

* Allow a specific port
```bash
firewall-cmd --permanent --add-port=8080/tcp
```

* Remove a port
```bash
firewall-cmd --permanent --remove-port=8080/tcp
```

* Reload to apply permanent changes to the running config
```bash
firewall-cmd --reload
```

>Without `--permanent`, changes apply only to the running configuration and are lost on reboot.

---

### AWS Security Groups

In AWS, Security Groups function as virtual firewalls attached to EC2 instances and other resources. Key characteristics:

- Stateful: you define inbound rules; return traffic is automatically allowed.
- Default behavior: all inbound traffic is denied; all outbound traffic is allowed.
- You add explicit inbound allow rules by protocol, port range, and source (CIDR or another security group).
- Multiple security groups can be attached to one instance.

Example inbound rules for a web server:

| Type   | Protocol | Port Range | Source        | Purpose             |
|--------|----------|------------|---------------|---------------------|
| SSH    | TCP      | 22         | Your office IP /32 | Admin access   |
| HTTP   | TCP      | 80         | 0.0.0.0/0    | Public web traffic  |
| HTTPS  | TCP      | 443        | 0.0.0.0/0    | Public HTTPS traffic |

---

**Security Groups vs. NACLs:** Security Groups operate at the instance level and are stateful. NACLs (Network Access Control Lists) operate at the subnet level and are stateless: you must explicitly allow both inbound and outbound directions. For most DevOps work, Security Groups are the primary tool. NACLs add an extra layer of subnet-level control when needed.

---

### `iptables`

`iptables` is the lower level Linux firewall framework that `firewalld` builds on top of. On newer systems, `iptables` commands are translated to `nftables` rules. You will encounter `iptables` directly on older systems and in Docker networking, where Docker adds its own rules to manage container networking.

Key concepts:

- **Tables:** `filter` (controls packet acceptance), `nat` (address translation), `mangle` (modifies packet headers).

- **Chains in the filter table:** `INPUT` (traffic destined for this host), `OUTPUT` (traffic originating from this host), `FORWARD` (traffic passing through this host).

- **Policy:** the default action if no rule matches (`ACCEPT` or `DROP`).

* List all rules with line numbers
```bash
iptables -L -n -v --line-numbers
```

* Allow incoming SSH
```bash
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
```

* Drop all other incoming traffic
```bash
iptables -P INPUT DROP
```

`iptables` rules are not persistent by default. Save and restore them with `iptables-save` and `iptables-restore`, or use `firewalld` or `ufw` which handle persistence automatically.

## 6. NAT and IP Addressing

### Private vs. Public IP Addresses

Not all IP addresses are reachable from the public internet. RFC 1918 defines three ranges reserved for private use:

| Range                                 | CIDR             | Addresses    | Common Use                                      |
|---------------------------------------|------------------|--------------|-------------------------------------------------|
| 10.0.0.0 to 10.255.255.255            | 10.0.0.0/8       | ~16.7 million | Large enterprise networks, AWS VPCs             |
| 172.16.0.0 to 172.31.255.255          | 172.16.0.0/12    | ~1 million    | Medium networks, Docker default bridge network  |
| 192.168.0.0 to 192.168.255.255        | 192.168.0.0/16   | ~65,000       | Home and small office networks, lab VMs         |

Private addresses can be reused freely across different organizations because they are not routed on the public internet. Two different companies can both use `10.0.0.0/8` internally without conflict.

Public IP addresses are globally unique and routable on the Internet. They are allocated by Regional Internet Registries (ARIN for North America, RIPE for Europe, APNIC for Asia-Pacific).

---

### Network Address Translation (NAT)

NAT lets devices with private IP addresses communicate with the public internet by translating their private source IP to a public IP at the boundary router or NAT gateway.

**How outbound NAT works:**

1. Your VM (`192.168.1.10`) sends a packet to `8.8.8.8:53` (Google DNS).
2. The packet reaches the router/NAT gateway.
3. The router replaces the source address with its own public IP (`203.0.113.5`) and records the mapping in a translation table.
4. Google receives the packet from `203.0.113.5` and responds to it.
5. The router receives the response, looks up the translation table, and forwards the packet to `192.168.1.10`.

The private device is completely hidden from the outside. This is why a home router can connect dozens of devices using a single public IP.

---

### NAT in AWS

AWS uses NAT extensively in VPC design:

**Internet Gateway (IGW):** Allows instances in public subnets to communicate directly with the internet. For IPv4 traffic, the IGW performs one to one NAT between the instance's private IP and its associated public IP or Elastic IP. Inbound connections from the internet to instances with public IPs are also possible through the IGW.

**NAT Gateway:** Allows instances in private subnets to initiate outbound connections to the internet (to download packages or call external APIs) without being directly reachable from the internet. The NAT Gateway sits in a public subnet and handles the address translation. The internet cannot initiate connections to private instances through a NAT Gateway.

A typical production VPC architecture:

```
Public Subnet (10.0.1.0/24)
- Web servers
- Load Balancer
- NAT Gateway

        |

Private Subnet (10.0.2.0/24)
- Application servers

        |

Private Subnet (10.0.3.0/24)
- Databases (Multi-AZ / second AZ for Amazon RDS)
```

Traffic flow:
- Internet > Load Balancer (public subnet) > App servers (private subnet)
- App servers > NAT Gateway (public subnet) > Internet (outbound only)
- No direct internet access to app servers or databases

---

### Special Purpose Addresses

| Range             | Purpose                                                                               |
|-------------------|---------------------------------------------------------------------------------------|
| 127.0.0.0/8       | Loopback (localhost). `127.0.0.1` always refers to the local machine.                |
| 169.254.0.0/16    | Link-local. Auto-assigned when DHCP fails. In AWS, `169.254.169.254` is the instance metadata endpoint (IMDS). |
| 0.0.0.0/0         | Represents all addresses. Used in routing tables (default route) and firewall rules.  |
| 255.255.255.255   | Limited broadcast. Sent to all hosts on the local network.                            |

---

**AWS Instance Metadata Service (IMDS):** The endpoint at `169.254.169.254` is how an EC2 instance retrieves its own metadata: instance ID, region, IAM role credentials, and more. AWS now defaults to IMDSv2, which requires a session-oriented request. You must first obtain a token with a PUT request, then use that token in subsequent GET requests:

* Get a session token (valid for 6 hours)
```bash
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
```

* Use the token to retrieve metadata
```bash
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id
```

When Terraform or the AWS CLI runs on an EC2 instance with an attached IAM role, it queries this endpoint automatically to get temporary credentials. Understanding IMDSv2 matters for both operations and security: tools running on EC2 need to support the session-oriented flow.

## 7. DHCP

DHCP (Dynamic Host Configuration Protocol) is what automatically assigns IP addresses to devices when they join a network. Without DHCP, every device would need a manually configured static IP, which is unmanageable at scale.

When a Linux server or container boots, it sends a broadcast request (DHCP Discover) looking for a DHCP server. The server responds with an offer (IP address, subnet mask, default gateway, DNS server, and lease duration). The client accepts the offer and uses those settings until the lease expires.

In AWS, every EC2 instance automatically receives an IP address, subnet mask, default gateway, and DNS resolver via DHCP. You do not configure this manually. The VPC's DHCP options set controls which DNS servers and domain names are handed out.

In Linux, you can see DHCP-assigned addresses with `ip addr show`. The lease information is stored in files like `/var/lib/dhclient/dhclient.leases` or managed by NetworkManager/systemd-networkd depending on the distribution.

**Why does this matter in DevOps?**

- Containers and VMs in most environments get their networking via DHCP.

- If DHCP fails, devices fall back to link local addresses in the `169.254.0.0/16` range. If you see a `169.254.x.x` address on an interface that should have a real IP, DHCP has failed.

- When designing server infrastructure, critical services (databases, load balancers, DNS servers) should always have static IPs or use DHCP reservations, not floating leases.

## 8. TLS and HTTPS

TLS (Transport Layer Security) is the protocol that encrypts traffic between a client and a server. It operates at the boundary between Layer 6 (Presentation) and Layer 7 (Application) in the OSI model. HTTPS is HTTP running over TLS.

As a DevOps engineer, you deal with TLS constantly: generating and renewing certificates, configuring Nginx to terminate TLS, debugging handshake failures, and setting up tools like Let's Encrypt or AWS Certificate Manager.

**How a TLS handshake works:**

1. The client connects to the server and sends the TLS versions and cipher suites it supports.
2. The server responds with its chosen cipher suite and its TLS certificate.
3. The client verifies the certificate against trusted Certificate Authorities (CAs).
4. They agree on a session key using asymmetric encryption (typically RSA or ECDHE).
5. All subsequent traffic is encrypted with that session key using symmetric encryption (typically AES).

**Major terms:**

- **Certificate:** A file containing the server's public key and identity, signed by a CA.

- **CA (Certificate Authority):** A trusted third party that signs certificates (Let's Encrypt, DigiCert, AWS ACM).

- **Self signed certificate:** A certificate signed by its own private key, not by a CA. Browsers and tools will warn or refuse to connect unless you explicitly trust it. Used in lab environments.

- **Certificate chain:** Many certificates are not signed directly by a root CA but by an intermediate CA. The full chain (leaf > intermediate > root) must be presented to clients.

- **SNI (Server Name Indication):** A TLS extension that lets a single server serve multiple certificates on one IP address. The client sends the hostname in the TLS handshake before encryption begins.

**Nginx TLS configuration (basic example):**

```nginx
server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate     /etc/nginx/ssl/example.com.crt;
    ssl_certificate_key /etc/nginx/ssl/example.com.key;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
}
```

TLSv1.0 and TLSv1.1 are deprecated and disabled in modern configurations. Only TLSv1.2 and TLSv1.3 should be used in production.

You can check the TLS configuration of any server with:

* View the certificate and negotiated protocol
```bash
curl -vI https://example.com 2>&1 | grep -E "(TLS|SSL|issuer|expire)"
```

* More detailed TLS inspection
```bash
openssl s_client -connect example.com:443 -servername example.com
```

## Lab 2.1.A: Network Diagnostics

**Objective:** Use standard Linux networking tools to inspect connectivity, routing, open ports, and DNS resolution. These tools are the first line of diagnosis for any network-related failure in a DevOps environment.

**Prerequisites:** A Linux VM running Fedora, CentOS Stream, or RHEL 9 with network access.

### Tool 1: `ping`

`ping` sends ICMP Echo Request packets to a host and reports whether it responds. It tests basic reachability at Layer 3.

* Basic ping: send 4 packets and stop
```bash
ping -c 4 8.8.8.8
```

* Ping a hostname (tests DNS resolution + reachability)
```bash
ping -c 4 google.com
```

* Ping with a specific packet size (tests MTU issues)
```bash
ping -c 4 -s 1400 8.8.8.8
```


Expected output:

**What the output tells you:**
  - `icmp_seq`: sequence number. Gaps indicate dropped packets.
  - `ttl`: Time to Live. Lower TTL suggests more hops to the destination.
  - `time`: round trip latency in milliseconds.
  - `packet loss`: any percentage above 0% indicates a network problem.

**Diagnostic scenarios:**
  - `ping 8.8.8.8` succeeds but `ping google.com` fails: DNS resolution is broken, not connectivity.
  - `ping 192.168.1.1` (your gateway) fails: a local network or firewall issue, not an internet issue.
  - 100% packet loss: the host is unreachable, a firewall is blocking ICMP, or the host does not exist.

---
>Note: Some hosts block ICMP even when services on them are perfectly reachable. A failed `ping` does not conclusively prove a host is down. Follow up with `curl` or `telnet` on a specific port.

## Lab 2.1.B: Subnet Calculation


**Objective:** Practice binary subnet calculation so you can correctly design IP address ranges for VPCs, subnets, and firewall rules without relying on a calculator.


**Background**

Every DevOps engineer who works with cloud infrastructure needs to understand subnetting. When you design an AWS VPC, you assign CIDR blocks to the VPC and to each subnet. If you get the ranges wrong — overlapping subnets, insufficient host capacity, or wrong broadcast addresses — the network will not route correctly.


### Full Worked Example: 192.168.1.0/26

**Step 1: Determine the subnet mask**

The prefix `/26` means 26 bits are the network portion. The remaining 32 - 26 = 6 bits are the host portion.

```
11111111 . 11111111 . 11111111 . 11000000
255      . 255      . 255      . 192
```

Subnet mask: **255.255.255.192**

**Step 2: Determine the block size**

Block size = 2^(host bits) = 2^6 = 64. Subnets with a /26 prefix start on multiples of 64: .0, .64, .128, .192.

**Step 3: Network address**

The network address is the first address in the block, with all host bits set to 0.

Network address: **192.168.1.0**

**Step 4: Broadcast address**

The broadcast address is the last address, with all host bits set to 1.

```
Last octet of 192.168.1.0 in binary: 00000000
Set all 6 host bits to 1:            00111111 = 63
```

Broadcast address: **192.168.1.63**

**Step 5: Usable host range**

- First usable: 192.168.1.1 (network address + 1)
- Last usable: 192.168.1.62 (broadcast address - 1)

**Step 6: Count usable hosts**

2^6 - 2 = **62 hosts**

**Final summary:**

| Property              | Value              |
|-----------------------|--------------------|
| Network               | 192.168.1.0/26     |
| Prefix length         | /26                |
| Subnet mask           | 255.255.255.192    |
| Network address       | 192.168.1.0        |
| Broadcast address     | 192.168.1.63       |
| First usable host     | 192.168.1.1        |
| Last usable host      | 192.168.1.62       |
| Number of usable hosts| 62                 |

## Lab 2.1.C: Static Route Configuration

**Objective:** Configure a static route between two VMs on different subnets and verify connectivity. This simulates routing between isolated network segments, a skill directly applicable to VPC routing tables and multi-tier architectures.

**Prerequisites:**
- Two Linux VMs in the same hypervisor (VMware Workstation or VirtualBox).
- Each VM has two network interfaces: one on a shared management network, one on a private network for the routing exercise.

**Lab Network Design:**

| Host | Interface         | IP Address       | Role                       |
|------|-------------------|------------------|----------------------------|
| VM1  | ens192 (management) | 192.168.1.10/24 | Source host               |
| VM1  | ens224 (private)    | 10.0.1.1/24     | Host in Subnet A          |
| VM2  | ens192 (management) | 192.168.1.11/24 | Destination host          |
| VM2  | ens224 (private)    | 10.0.1.254/24 and 10.0.2.1/24 | Acts as router between subnets |

The goal: make VM1 (10.0.1.1) able to reach VM2's address in Subnet B (10.0.2.1) via the private interfaces.

---

### Step 1: Assign IP Addresses to the Private Interfaces

* On VM1
```bash
ip addr add 10.0.1.1/24 dev ens224
ip link set ens224 up

- Verify
```bash
ip addr show ens224
```

* On VM2
```bash
ip addr add 10.0.1.254/24 dev ens224
ip addr add 10.0.2.1/24 dev ens224
ip link set ens224 up
```
- Verify
```bash
ip addr show ens224
```

>Note: `ip addr add` is not persistent. These addresses disappear on reboot. For persistence on RHEL/Fedora, use `nmcli` (covered in Section 2.2). For this lab, temporary assignment is fine.

---

### Step 2: Enable IP Forwarding on VM2

VM2 needs to forward packets between the two subnets. By default, Linux does not forward packets between interfaces.

* On VM2: enable IP forwarding
```bash
echo 1 > /proc/sys/net/ipv4/ip_forward
```
* Verify
```bash
cat /proc/sys/net/ipv4/ip_forward   # Should return 1
```

* Make it persistent across reboots
```bash
echo "net.ipv4.ip_forward = 1" >> /etc/sysctl.conf
```
```bash
sysctl -p
```

---

### Step 3: Check the Routing Table Before Adding Routes

On both VMs, view the current routing table:

```bash
ip route show
```

Expected output on VM1 (before adding the static route):
```
default via 192.168.1.1 dev ens192 proto static metric 100
192.168.1.0/24 dev ens192 proto kernel scope link src 192.168.1.10
10.0.1.0/24 dev ens224 proto kernel scope link src 10.0.1.1
```

VM1 knows about its own subnets but has no route to 10.0.2.0/24. A packet from VM1 to 10.0.2.1 would hit the default gateway and fail.

**Save this output as your "before" evidence.**

---

### Step 4: Add the Static Route

On VM1, add a route to Subnet B via VM2's Subnet A address:

```bash
ip route add 10.0.2.0/24 via 10.0.1.254 dev ens224
```

---

### Step 5: Check the Routing Table After Adding the Route

On VM1:
```bash
ip route show
```

Expected output:
```
default via 192.168.1.1 dev ens192 proto static metric 100
192.168.1.0/24 dev ens192 proto kernel scope link src 192.168.1.10
10.0.1.0/24 dev ens224 proto kernel scope link src 10.0.1.1
10.0.2.0/24 via 10.0.1.254 dev ens224
```

The new line `10.0.2.0/24 via 10.0.1.254 dev ens224` confirms the static route is in place.

**Save this output as your "after" evidence.**

---

### Step 6: Verify Connectivity

From VM1:
```bash
ping -c 4 10.0.2.1
```
- Expected successful output. If ping fails, check in this order:
1. Is the route present? (`ip route show` on VM1)
2. Is IP forwarding enabled on VM2? (`cat /proc/sys/net/ipv4/ip_forward` should return `1`)
3. Is a firewall on VM2 dropping forwarded packets? (`firewall-cmd --list-all` or `iptables -L FORWARD`)
4. Does VM2 have both IPs on ens224? (`ip addr show ens224`)

---

### Step 7: Confirm the Path with `traceroute`

From VM1:
```bash
traceroute 10.0.2.1
```

Expected output:
```
traceroute to 10.0.2.1 (10.0.2.1), 30 hops max, 60 byte packets
 1  10.0.1.254 (10.0.1.254)  0.5 ms  0.4 ms  0.4 ms
 2  10.0.2.1 (10.0.2.1)      0.8 ms  0.7 ms  0.7 ms
```

Hop 1 is VM2 acting as the router. Hop 2 is the destination. The static route is working.
