## Load Balancer

This guide covers the fundamentals, architecture, configuration, and advanced options for setting up Nginx as a load balancer in DevOps environments.

---

**What is a Load Balancer?**

A load balancer is a software or hardware component that distributes incoming network traffic (HTTP/S, TCP, UDP) across multiple backend servers in a pool. Its primary goal is to prevent any single server from becoming a bottleneck.

- **Uses:**
  - Websites & web applications (HTTP/HTTPS)
  - API endpoints (REST, GraphQL, etc.)
  - Database read replicas
  - Network services (DNS, FTP, SMTP, VPN)
  - Microservices architectures (internal/external routing)
  - Traffic spike management (flash sales, viral content)
  - SSL/TLS offloading (encryption handled at the load balancer)

- **Advantages:**
  - **Improved Reliability:** Keeps applications available even when individual servers fail.
  - **Enhanced Performance:** Reduces latency and distributes server load more evenly.
  - **Increased Scalability:** New servers can be added to the pool without disrupting users.

| Function | Description |
| :--- | :--- |
| **Traffic Distribution** | Efficiently directs incoming client requests to available servers. |
| **High Availability** | Automatically detects server failures and redirects traffic to healthy servers, preventing downtime. |
| **Scalability** | Enables seamless addition of servers to the backend pool without impacting users. |
| **Performance Optimization** | Ensures requests are handled by the most available or capable server, reducing load and latency. |

---

**Load Balancing Algorithms**

The algorithm determines how the load balancer distributes traffic across backend servers.

| Algorithm | Principle | Best For |
| :--- | :--- | :--- |
| **Round Robin** | Distributes requests sequentially to each server in a loop. | Uniform servers with similar performance. |
| **Least Connections** | Directs traffic to the server with the fewest active connections. | Long-lived connections (e.g., databases, live streams). |
| **Least Response Time** | Routes to the server with the lowest average response time and fewest connections. *Note: Nginx Plus only (`least_time` directive); not available in open source Nginx.* | Performance critical applications with low latency requirements. |
| **IP Hash** | Uses the client's IP address to pin them to a specific server, ensuring session persistence. | Session-stateful workloads (e.g., shopping carts, user logins). |

**Load Balancer Request Flow**

A typical request cycle works like this:
1. A user sends a request (e.g., to visit a website).
2. The load balancer receives the request.
3. The load balancer applies a configured algorithm to select the optimal backend server.
4. The request is forwarded to the chosen server.
5. The server processes the request and returns a response, typically through the load balancer.

---

**Types of Load Balancers** Categorized by form factor and deployment model

- **Hardware Load Balancers**
  - Dedicated physical appliances.
  - High performance and capable of handling massive traffic volumes.
  - Expensive and less flexible than software alternatives.

- **Software Load Balancers**
  - Applications running on standard servers or VMs.
  - More flexible and cost effective.
  - Requires management of the underlying OS.

- **Cloud Load Balancers**
  - Managed services from cloud providers (e.g., AWS ELB, Azure Load Balancer, GCP Cloud Load Balancing).
  - Handle scaling and maintenance automatically, with high availability built in.
  - Can be vendor specific.

**SELinux (Security Enhanced Linux)**

SELinux provides mandatory access control (MAC) for fine-grained system security.

- **Enforcing:** SELinux policies are active and enforced.
- **Permissive:** Violations are logged but not blocked,   useful for troubleshooting.
- **Disabled:** No enforcement at all.

**Commands:**
```bash
getenforce                                         # Check current SELinux mode

getsebool -a                                       # List all SELinux booleans

setsebool -P httpd_can_network_connect=1           # Allow Nginx to make outbound network connections (persistent)
getsebool -a | grep httpd_can_network_connect      # Verify the setting
```

**Load Balancer Setup**

**Step 1: Install Nginx on the Load Balancer**

```bash
# RHEL/CentOS/Fedora
sudo yum install epel-release -y
sudo yum install nginx -y

# Debian/Ubuntu
sudo apt update
sudo apt install nginx -y

# Start and enable Nginx
sudo systemctl start nginx
sudo systemctl enable nginx
sudo systemctl status nginx
```

**Step 2: Configure the Firewall**

```bash
# Open HTTP and HTTPS ports using service names
sudo firewall-cmd --permanent --add-service=http
sudo firewall-cmd --permanent --add-service=https
sudo firewall-cmd --reload

# Alternative: Open specific port numbers
sudo firewall-cmd --permanent --add-port=80/tcp
sudo firewall-cmd --permanent --add-port=443/tcp
sudo firewall-cmd --reload
```

**Step 3: Configure Load Balancing**

Create the config file: `sudo vim /etc/nginx/conf.d/loadbalancer.conf`

```nginx
upstream backend_servers {
    # Round-robin is the default   no directive needed
    server 10.10.0.7;
    server 10.10.0.8;
}

server {
    listen 80;
    server_name 10.10.0.6;

    access_log /var/log/nginx/lb_access.log;
    error_log /var/log/nginx/lb_error.log;

    location / {
        proxy_pass http://backend_servers;

        # Essential proxy headers
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Timeout settings
        proxy_connect_timeout 30s;
        proxy_send_timeout 30s;
        proxy_read_timeout 30s;
    }
}
```

Check syntax and restart:
```bash
sudo nginx -t

sudo systemctl restart nginx
```

**Load Balancing Methods**

**A. Round Robin (default)**
Distributes requests sequentially across all servers in rotation. No directive is needed   this is the default behavior.

```nginx
# Syntax
upstream backend_name {
    server host:port;
    server host:port;
}

# Example
upstream backend {
    server 192.168.1.10;
    server 192.168.1.11;
}
```

**B. Least Connections**
Routes each new request to the server with the fewest active connections.

```nginx
# Syntax
upstream backend_name {
    least_conn;
    server host:port;
}

# Example
upstream backend {
    least_conn;
    server 192.168.1.10;
    server 192.168.1.11;
}
```

**C. IP Hash (Session Persistence)**
Uses the client's IP address to consistently route them to the same backend server.

```nginx
# Syntax
upstream backend_name {
    ip_hash;
    server host:port;
}

# Example
upstream backend {
    ip_hash;
    server 192.168.1.10;
    server 192.168.1.11;
}
```

**D. Weighted Load Balancing**
Assigns a relative weight to each server; higher-weight servers receive proportionally more traffic.

```nginx
# Syntax
upstream backend_name {
    server host:port weight=number;
    server host:port weight=number;
}

# Example
upstream backend {
    server 192.168.64.6:80 weight=3;    # Receives ~60% of traffic
    server 192.168.64.7:80 weight=2;    # Receives ~40% of traffic
}
```

**E. Generic Hash**
Routes requests based on a hashed key (e.g., the request URI), ensuring consistent routing for the same key.

```nginx
# Syntax
upstream backend_name {
    hash key;
    server host:port;
}

# Example
upstream backend {
    hash $request_uri;
    server 192.168.1.10;
    server 192.168.1.11;
}
```

**F. Random with Two Choices**
Randomly selects two servers from the pool, then forwards the request to whichever has fewer active connections. Available in open-source Nginx since version 1.15.1.

```nginx
# Syntax
upstream backend_name {
    random two;
    server host:port;
}

# Example
upstream backend {
    random two;
    server 192.168.1.10;
    server 192.168.1.11;
}
```

**Health Checks and Failover Configuration**

```nginx
upstream backend_name {
    server host:port max_fails=number fail_timeout=time;
    server host:port max_fails=number fail_timeout=time;
    server host:port backup;
}
```

> **Note:** The parameters below configure *passive* health checks, which are available in open source Nginx. *Active* health checks (periodic probes sent to backends regardless of live traffic) require **Nginx Plus**.

**Server Health Parameters:**
- `max_fails=number`   Number of consecutive failures before a server is marked unhealthy.
- `fail_timeout=time`   How long a server remains marked as failed before being retried (e.g., `30s`).
- `backup`   Marks the server as a standby; only used when all primary servers are unavailable.

---

**Enhanced Health Check Directives**

```nginx
location /path {
    proxy_pass http://upstream_name;

    # Conditions that trigger a retry on the next server
    proxy_next_upstream error_type [error_type...];

    # Retry limits
    proxy_next_upstream_tries number;
    proxy_next_upstream_timeout time;

    # Essential headers for the backend
    proxy_set_header Header_Name $variable;
}
```

**Health Check Parameters:**
- `proxy_next_upstream`   Conditions under which Nginx retries the next server:
  - `error`   Connection or communication error.
  - `timeout`   A timeout occurred.
  - `http_500`   Backend returned HTTP 500.
  - `http_502`   Backend returned HTTP 502.
  - `http_503`   Backend returned HTTP 503.
  - `http_504`   Backend returned HTTP 504.
- `proxy_next_upstream_tries`   Maximum number of upstream servers to attempt.
- `proxy_next_upstream_timeout`   Time limit for all retry attempts combined.

**Essential Headers:**
- `Host $host`   Preserves the original request host header.
- `X-Real-IP $remote_addr`   Forwards the direct client IP to the backend.
- `X-Forwarded-For $proxy_add_x_forwarded_for`   Appends the client IP to any existing forwarding chain (do not use `$remote_addr` here   it would overwrite the chain).

**Load Balancer Monitoring**

```nginx
location /monitoring_path {
    stub_status on;
    access_log off;
    allow network;
    deny all;
}
```

**Monitoring Directives:**
- `stub_status on`   Enables the built-in Nginx status page.
- `access_log off`   Suppresses log entries for the status endpoint.
- `allow / deny`   Restricts access to trusted networks only.

**Status Page Provides:**
- Number of active connections.
- Total accepted, handled, and current requests.
- Reading, writing, and waiting connection states.

**Backup Server**

A backup server remains idle during normal operation and is only activated when all primary servers are unavailable. It automatically returns to standby once a primary server recovers.

```nginx
server host:port backup;
```

**Example Configuration**

```nginx
# /etc/nginx/conf.d/loadbalancer.conf
upstream backend {
    least_conn;                                              # Use least connections method
    server 192.168.1.10:80 max_fails=3 fail_timeout=30s;   # Primary server with passive health check
    server 192.168.1.11:80;                                 # Secondary primary server
    server 192.168.1.12:80 backup;                          # Backup   only used if both primaries fail
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;   # Correct: preserves IP chain
    }
}
```

**Enterprise & Cloud-Native Enhancements**

**A. Dynamic Backend Registration (Service Discovery)**

Instead of maintaining static backend lists, service discovery tools can update Nginx's upstream configuration automatically.

- **Service Discovery Integration:**
  - Consul, etcd, or the Kubernetes API for real-time node registration.
  - Nginx Ingress Controller automatically watches Kubernetes endpoint changes.

- **DNS-Based Discovery:**
```nginx
upstream backend {
    server backend-service.internal.com;   # Resolved at startup; use DNS TTL and resolver for dynamic re-resolution

    resolver 10.0.0.2 valid=30s;           # DNS server with cache TTL
    resolver_timeout 5s;                   # DNS query timeout
}
```

> **Note:** The `resolve` parameter (to continuously re-resolve DNS at runtime) is an **Nginx Plus** feature. In open source Nginx, the hostname is resolved only at startup or config reload.

- **Template-Based Automation (Consul Template):**
```nginx
upstream backend {
    {% for server in service "web" %}
    server {{ server.Address }}:{{ server.Port }};
    {% endfor %}
}
```

- **Advantages:**
  - New nodes join the pool automatically.
  - Failed nodes are removed without manual intervention.
  - Zero downtime updates with dynamic backend changes.
  - Integrates naturally with container orchestration platforms.

---

**B. Automated Blue/Green & Canary Deployments**

These strategies minimize risk and enable zero-downtime releases.

- **Blue/Green:**
  Deploy the new version alongside the current one, validate it, then switch all traffic over.

```nginx
upstream backend {
    server blue-environment:80;    # Current stable version (receiving all traffic)
    server green-environment:80;   # New version (staged, ready to activate)
}
```

- **Canary:**
  Gradually shift a small percentage of traffic to the new version, monitor metrics, then increase the proportion.

```nginx
upstream backend {
    server stable-version:80 weight=90;    # 90% to stable
    server canary-version:80 weight=10;    # 10% to new version
}
```

- **Session Based Canary with Consistent Hashing:**
  Ensures a given user consistently hits the same backend during the canary period.

```nginx
upstream backend {
    hash $cookie_canary consistent;    # Consistent hashing for session affinity
    server stable-version:80;
    server canary-version:80;
}
```

- **Automated Rollback:** Traffic automatically reverts to healthy servers if health checks fail.

- **Implementation Tools:**
  - **Kubernetes:** Labels, selectors, and service meshes.
  - **Orchestration:** Argo Rollouts, Flagger for automated traffic shifting.
  - **CI/CD Integration:** Automated promotion based on metrics and health check results.

- **Advantages:**
  - Zero-downtime deployments.
  - Reduced risk through gradual traffic exposure.
  - Instant rollback capability.
  - Real time monitoring during releases.

**F. Advanced Traffic Shaping & Rate Limiting**

Control traffic flow using Nginx's built in modules to prevent abuse and maintain service stability.

**1. Rate Limiting Configuration**

```nginx
# Define a rate-limit zone: max 10 requests/second per IP (10MB shared memory)
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

# Define a connection-limit zone
limit_conn_zone $binary_remote_addr zone=addr:10m;

server {
    location /api/ {
        limit_req zone=api burst=20 nodelay;    # Allow bursting up to 20; process without delay
        limit_conn addr 10;                     # Max 10 concurrent connections per IP
        proxy_pass http://backend;
    }
}
```

**Parameters Explained:**
- `zone=name:size`   Shared memory zone for tracking state across workers.
- `rate=requests`   Maximum request rate (`r/s` = per second, `r/m` = per minute).
- `burst=number`   Number of excess requests that can be queued before being rejected.
- `nodelay`   Queued burst requests are processed immediately rather than throttled.
- `limit_conn`   Caps concurrent connections per key (e.g., per IP).

---

**2. Advanced Traffic Management**

- **Selective Rate Limiting:**
```nginx
location /api/public/ {
    limit_req zone=public_api burst=50 nodelay;    # More lenient for public endpoints
}

location /api/private/ {
    limit_req zone=private_api burst=10 nodelay;   # Stricter for sensitive endpoints
}

location /static/ {
    proxy_pass http://backend;                     # No rate limiting for static assets
}
```

- **Geographic or User Based Limits:**
```nginx
geo $limited_region {
    default 0;
    192.168.1.0/24 1;    # Internal network   bypass rate limiting
}

map $limited_region $limit_key {
    0 $binary_remote_addr;    # External users   limit by IP
    1 "";                     # Internal users   no limit
}

limit_req_zone $limit_key zone=geo_aware:10m rate=5r/s;
```

**4. Benefits of Traffic Shaping**

- **Service Protection:** Prevents resource exhaustion and absorbs DDoS traffic.
- **Fair Usage:** Ensures equitable resource distribution across users.
- **Quality of Service:** Maintains performance stability during traffic spikes.
- **Cost Control:** Reduces infrastructure costs by eliminating abusive traffic.
- **Compliance:** Supports API rate limiting requirements and SLA enforcement.

---

**G. Global Load Balancing (GeoDNS, Anycast)**

- Deploy GeoDNS to route users to the nearest datacenter or regional load balancer.
- Use Anycast IP addressing for automatic global failover and geographic distribution.
- Integrate with cloud DNS services (AWS Route 53, Azure Traffic Manager, Google Cloud DNS) for policy based global routing.
- Example:
  - A GeoDNS record for `app.example.com` resolves to different regional Nginx load balancers depending on the user's location.
  - Each regional LB handles local traffic and can failover to another region if needed.

**Recommendations**

- **Monitor Nginx and Backend Health:** Use the Prometheus Nginx exporter with Grafana dashboards, and review access/error logs regularly.
- **Automate with IaC:** Use Ansible or Terraform to manage Nginx configuration and server provisioning consistently.
- **SSL/TLS Offloading:** Terminate HTTPS at the load balancer to centralize certificate management and reduce backend overhead.
- **Web Application Firewall (WAF):** Integrate ModSecurity or a cloud WAF with Nginx for additional security.
- **Containerize the Load Balancer:** Run Nginx as a container or deploy it in Kubernetes with Ingress controllers for dynamic scaling.
- **CI/CD Integration:** Automate config validation (`nginx -t`) and reloads via deployment pipelines.
- **SELinux:** Always verify and configure the necessary booleans (e.g., `httpd_can_network_connect`) when running Nginx on SELinux enforced systems.
