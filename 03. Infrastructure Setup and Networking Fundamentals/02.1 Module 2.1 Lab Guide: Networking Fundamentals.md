## Lab 2.1.A: Network Diagnostics

### Objective

Use the standard Linux networking toolkit, `ping`, `traceroute`, `ss`, `nslookup`, and `dig`, to diagnose a deliberately broken DNS and routing scenario. By the end, you will have practiced the bottom up diagnostic habit from the study note on a real, broken system instead of a healthy one, which is where the habit actually earns its value.

### What You Need

- A Linux VM (Fedora, CentOS Stream, or RHEL 9) with network access
- Sudo access on the VM

### Prerequisites

- Read Section 2.1 of this module, specifically the OSI model and TCP/IP sections
- A working VM from Module 2's virtualization setup

---
### Part 1: Build Your Diagnostic Toolkit

Before breaking anything, confirm every tool you need is installed and working.

- Confirm each tool is present
```bash
which ping traceroute ss nslookup dig
```
- Install anything missing
```bash
sudo dnf install -y traceroute bind-utils
```

`ping` and `ss` ship with the base system. `traceroute` and the DNS tools (`nslookup`, `dig`) come from separate packages on Fedora and RHEL family systems.

---
### Part 2: Establish a Healthy Baseline

You cannot recognize broken behavior if you have never seen working behavior on the same machine. Run each tool once against a known good target and record the output before you touch any configuration.

- Layer 3: basic reachability
```bash
ping -c 4 8.8.8.8
```
- Layer 3 with name resolution involved
```bash
ping -c 4 google.com
```
- The path packets take to reach a known good destination
```bash
traceroute 8.8.8.8
```
- What is currently being listened to on this machine
```bash
ss -tlnp
```
- Basic DNS resolution is working normally
```bash
nslookup google.com
```
- Detailed DNS resolution is working normally
```bash
dig google.com A
```

For each command, write one sentence in your own words describing what a healthy result looks like. You will compare this baseline against the broken scenario in Part 4.

---
### Part 3: Break Your Own DNS Deliberately

You are about to deliberately misconfigure DNS resolution on your own VM so you can practice diagnosing it. This is safe to do in a lab VM. Do not do this on a production system.

- Back up your current DNS configuration first
```bash
sudo cp /etc/resolv.conf /etc/resolv.conf.backup
```

- Point DNS resolution at a nonexistent, unreachable resolver
```bash
sudo bash -c 'cat > /etc/resolv.conf << EOF
nameserver 192.0.2.1
EOF'
```

`192.0.2.0/24` is a reserved address block specifically set aside for documentation and testing. It is guaranteed to never be a real, reachable host, which makes it perfect for this exercise.

If your system uses NetworkManager and tends to overwrite `/etc/resolv.conf` automatically, also run:

- Temporarily stop NetworkManager from regenerating the file
```bash
sudo nmcli connection modify "$(nmcli -t -f NAME connection show --active | head -1)" ipv4.dns "192.0.2.1"
sudo nmcli connection modify "$(nmcli -t -f NAME connection show --active | head -1)" ipv4.ignore-auto-dns yes
sudo systemctl restart NetworkManager
```

---
### Part 4: Diagnose the Broken Scenario

Now work through the same bottom up sequence from the study note on the now broken system. Run each command, capture the output, and write what it reveals.

**Step 1, Layer 3 check:**

```bash
ping -c 4 8.8.8.8
```

This should still succeed. You broke DNS, not basic IP connectivity. Write one sentence confirming Layer 3 is healthy and explaining why this result alone does not yet tell you anything is wrong.

**Step 2, name resolution check:**

```bash
ping -c 4 google.com
```

This should fail or hang because `ping` cannot resolve the hostname to an IP address before it can even send the first packet. Write one sentence explaining specifically what this failure, combined with the success of Step 1, narrows the problem down to.

**Step 3, confirm with the dedicated DNS tools:**

```bash
nslookup google.com
```

```bash
dig google.com A
```

Both should report an error or simply time out, since they are trying to query a resolver at an address that does not exist. Capture the exact error message each tool produces.

**Step 4, query a different resolver directly to isolate the problem further:**

```bash
dig @8.8.8.8 google.com A
```

This bypasses your broken local configuration entirely and asks Google's resolver directly. It should succeed. Write two or three sentences explaining why this single command proves conclusively that the problem is specifically your local DNS configuration, and not a broader internet connectivity problem or a problem with Google's own DNS infrastructure.

---
### Part 5: Fix It and Verify

- Restore your original configuration
```bash
sudo cp /etc/resolv.conf.backup /etc/resolv.conf
```

If you modified NetworkManager settings in Part 3, also reverse those:

```bash
sudo nmcli connection modify "$(nmcli -t -f NAME connection show --active | head -1)" ipv4.ignore-auto-dns no
sudo systemctl restart NetworkManager
```

Confirm the fix:

```bash
nslookup google.com
```
```bash
dig google.com A
```
```bash
ping -c 4 google.com
```

All three should now succeed and match your healthy baseline from Part 2.

---
### Part 6: Diagnose a Routing Scenario

DNS is one category of failure. Routing is another. This part uses `traceroute` to diagnose a path problem rather than a name resolution problem.

- Trace the path to a real destination
```bash
traceroute 8.8.8.8
```
- Trace the path to your own default gateway
```bash
ip route show | grep default
```

Take the gateway address shown in that output and use it as the target for a second trace:

```bash
traceroute <paste the gateway address here>
```

Compare the two traces. The trace to your gateway should complete in a single hop. The trace to 8.8.8.8 should show several hops.

Now examine your routing table directly:

```bash
ip route show
```

Write two or three sentences explaining, in your own words, what the default route line in this output means, and what would happen to all of your outbound traffic if that line were deleted.

Document all of the following in a single write up:

1. Your healthy baseline output and one sentence per tool describing normal behavior
2. The exact steps you took to break DNS
3. Your full bottom up diagnosis of the broken scenario, with the reasoning written out at each step, not just the raw command output
4. The specific command and reasoning that proved the problem was local DNS configuration and not broader internet connectivity
5. Confirmation that you restored DNS and verified it works again
6. Your routing table output and an explanation of the default route

---
### Common Issues

**`/etc/resolv.conf` keeps reverting on its own.** On systems using NetworkManager, the file is often regenerated automatically. Use the `nmcli ipv4.ignore-auto-dns yes` approach from Part 3 if a plain file edit does not stick.

**`ping 8.8.8.8` also fails, not just `ping google.com`.** This means you have a Layer 3 problem on top of, or instead of, the DNS problem you intended to create. Check `ip addr show` to confirm your VM still has a valid IP address, and check `ip route show` to confirm a default route still exists.

**`traceroute` shows nothing but asterisks for every hop.** Some networks and firewalls block the ICMP or UDP probes traceroute relies on. Try `traceroute -I 8.8.8.8` to switch to ICMP probes, which get through more firewalls than the default UDP probes.

## Lab 2.1.B: Subnet Design

### Objective

Build real fluency with CIDR math by working a full subnet calculation by hand, then apply that fluency to design a realistic three tier VPC layout. By the end, you should be able to look at a CIDR block and immediately know its size without reaching for a calculator.

### What You Need

- A calculator or spreadsheet, used only to check your work, not to do it for you
- No VM or terminal required for this lab

### Prerequisites

Read Section 2.1 of this module, specifically the NAT, Addressing, and CIDR section.

---
### Part 1: Worked Example, Full Method

Work through `192.168.1.0/26` using the complete binary method. Do not skip steps, even the ones that feel obvious. The goal is to build the underlying mental model, not just memorize a shortcut formula.

**Step 1: Determine the subnet mask**

The prefix `/26` means 26 bits belong to the network portion. The remaining `32 - 26 = 6` bits belong to the host portion.

```
11111111 . 11111111 . 11111111 . 11000000
   255    .   255    .   255    .   192
```

Subnet mask: `255.255.255.192`

**Step 2: Determine the block size**

Block size equals 2 raised to the power of the host bits. `2^6 = 64`. This means valid subnets of this size start on multiples of 64: `.0`, `.64`, `.128`, `.192`.

**Step 3: Find the network address**

The network address is the very first address in the block, with every host bit set to 0.

Network address: `192.168.1.0`

**Step 4: Find the broadcast address**

The broadcast address is the very last address in the block, with every host bit set to 1.

```
Last octet of the network address in binary: 00000000
Set all 6 host bits to 1:                    00111111  =  63
```

Broadcast address: `192.168.1.63`

**Step 5: Determine the usable host range**

```
First usable: 192.168.1.1   (network address plus 1)
Last usable:  192.168.1.62  (broadcast address minus 1)
```

**Step 6: Count usable hosts**

`2^6 - 2 = 62`. You subtract 2 because the network address and the broadcast address both exist within the range but neither can be assigned to an actual device.

**Final summary table:**

| Property | Value |
|---|---|
| Network | 192.168.1.0/26 |
| Subnet mask | 255.255.255.192 |
| Network address | 192.168.1.0 |
| Broadcast address | 192.168.1.63 |
| First usable host | 192.168.1.1 |
| Last usable host | 192.168.1.62 |
| Usable hosts | 62 |

### Part 3: Design a Three Tier VPC

Now apply what you just practiced to a realistic infrastructure design problem.

**The scenario:** You are designing the network layout for a small production application with three tiers: a web tier, an application tier, and a database tier. The company has allocated you the address space `10.20.0.0/16` for this entire VPC.

**Your design requirements:**

1. The web tier needs to be reachable from the public internet and should have room for up to 50 hosts as the application grows
2. The application tier must never be directly reachable from the public internet, and should have room for up to 100 hosts
3. The database tier must never be directly reachable from the public internet, requires the strongest isolation of the three tiers, and only needs room for a small number of hosts, fewer than 10
4. You must design for two separate availability zones for resilience, meaning every tier needs a subnet in each of the two zones

**Your task:**

Design the complete subnet layout. For each subnet, specify the CIDR block, which tier and availability zone it belongs to, whether it is public or private, and the number of usable hosts it provides. Use this table:

| Subnet Name | CIDR Block | Tier | Availability Zone | Public or Private | Usable Hosts |
|---|---|---|---|---|---|
| | | Web | AZ-A | | |
| | | Web | AZ-B | | |
| | | App | AZ-A | | |
| | | App | AZ-B | | |
| | | Database | AZ-A | | |
| | | Database | AZ-B | | |

Choose CIDR blocks that comfortably meet each tier's stated host requirement without wasting excessive address space. Show the math for each subnet's usable host count to confirm it meets the requirement.

After completing the table, answer these three questions in writing:

1. Why does the database tier need its own separate subnet rather than simply sharing the application tier's subnet with stricter security group rules layered on top?

2. Your total `/16` allocation provides roughly 65,000 addresses. Add up the total addresses your six subnets actually consume. What does the large remaining gap tell you about how real organizations typically plan VPC address space, and why might that be intentional rather than wasteful?

3. If the company later needs to add a fourth tier, for example a caching layer, would your current design accommodate that without requiring you to redesign any of the existing subnets? Explain why or why not.

---
### Common Mistakes

**Choosing a subnet size by guessing instead of calculating.** Always calculate the usable host count for a candidate CIDR block before committing to it. A `/27` looks similar to a `/26` at a glance but holds half as many usable addresses.

**Forgetting AWS reserves five addresses per subnet.** If you are sizing a subnet specifically for AWS, a `/24` gives you 251 usable addresses in practice, not the 254 a standard, non cloud calculation would suggest. This rarely changes your overall design decision, but it is worth remembering when you size something close to its limit.

**Designing only for today's host count with zero room to grow.** A subnet sized exactly to your current need leaves no room for auto scaling or future expansion. Sizing generously up front, within reason, is far cheaper than a redesign later once real, dependent resources already exist inside that subnet.

## Lab 2.1.C: Static Routing

### Objective

Configure a static route between two VMs sitting on different subnets and watch connectivity go from failing to working as a direct, visible result of one single command. This makes the abstract idea of a routing table concrete by forcing you to build, break, and fix connectivity yourself rather than simply reading about how it works.

### What You Need

- Two Linux VMs on the same hypervisor, each with two network interfaces
- Sudo access on both VMs

### Prerequisites

- Read Section 2.1 of this module, specifically the OSI model and CIDR sections
- Two VMs cloned from your base image, following the cloning process from Module 2's virtualization labs
- Each VM must have a second network adapter attached, in addition to the management adapter you already use to SSH in

---
### Lab Network Design

| Host | Interface | IP Address | Role |
|---|---|---|---|
| VM1 | Management (existing) | Whatever you currently use to SSH in | Reaching the VM from your host machine |
| VM1 | Private (second adapter) | 10.0.1.1/24 | Source host in Subnet A |
| VM2 | Management (existing) | Whatever you currently use to SSH in | Reaching the VM from your host machine |
| VM2 | Private (second adapter) | 10.0.1.254/24 and 10.0.2.1/24 | Acts as the router between Subnet A and Subnet B |

The goal of this lab is to make VM1, sitting in Subnet A at `10.0.1.1`, able to reach an address in Subnet B, `10.0.2.1`, which only VM2 currently holds. VM1 has no way to reach Subnet B until you explicitly tell it how to get there.

Confirm both VMs have a second network adapter attached in your hypervisor settings before starting. If you only see one interface when you run `ip addr show`, stop and add the second adapter in VirtualBox or VMware first.

---
### Step 1: Identify Your Second Interface Name

Interface names vary by hypervisor and VM configuration. Find yours on each VM:

```bash
ip addr show
```

Look for an interface beyond `lo` (loopback) and your existing management interface. It might be named `ens224`, `eth1`, `enp0s8`, or something similar depending on your setup. Use the actual name you see for the rest of this lab, substituting it wherever this guide says `<private-iface>`.

---
### Step 2: Assign IP Addresses to the Private Interfaces

On VM1:

```bash
sudo ip addr add 10.0.1.1/24 dev <private-iface>
sudo ip link set <private-iface> up
ip addr show <private-iface>
```

On VM2:

```bash
sudo ip addr add 10.0.1.254/24 dev <private-iface>
sudo ip addr add 10.0.2.1/24 dev <private-iface>
sudo ip link set <private-iface> up
ip addr show <private-iface>
```

VM2 now holds two separate addresses on the same physical interface, one in each subnet. This is intentional. It is what lets VM2 act as the connection point between Subnet A and Subnet B.

These addresses are temporary and will not survive a reboot. That is fine for this lab. Persistent configuration through `nmcli` is covered separately in this module.

---
### Step 3: Enable IP Forwarding on VM2

By default, Linux does not forward packets from one interface to another. A packet arriving on one interface, destined for an address reachable only through a different interface, gets silently dropped unless forwarding is explicitly enabled.

On VM2 only:

```bash
# Enable forwarding immediately
sudo sysctl -w net.ipv4.ip_forward=1

# Confirm
sysctl net.ipv4.ip_forward
```

Expected output: `net.ipv4.ip_forward = 1`

```bash
# Make it persistent across reboots
echo "net.ipv4.ip_forward = 1" | sudo tee -a /etc/sysctl.conf
sudo sysctl -p
```

---
### Step 4: Capture the Routing Table Before Adding the Route

On VM1, before doing anything else:

```bash
ip route show
```

You should see something like:

```
default via <your-gateway> dev <management-iface> proto static metric 100
<your-management-subnet> dev <management-iface> proto kernel scope link
10.0.1.0/24 dev <private-iface> proto kernel scope link src 10.0.1.1
```

Notice what is missing. VM1 knows how to reach its own management network and its own private subnet, `10.0.1.0/24`, automatically, simply because it has an address directly on those networks. It has no entry at all for `10.0.2.0/24`. Save this exact output. This is your before evidence.

Try to confirm this gap experimentally:

```bash
ping -c 2 10.0.2.1
```

This should fail or hang. VM1 has no route telling it how to reach that address.

---
### Step 5: Add the Static Route

On VM1, add a route telling it that the way to reach Subnet B is through VM2's address in Subnet A:

```bash
sudo ip route add 10.0.2.0/24 via 10.0.1.254 dev <private-iface>
```

Read this command carefully and understand each part. You are telling VM1: to reach anything in `10.0.2.0/24`, send the packet via `10.0.1.254`, using the `<private-iface>` interface. `10.0.1.254` is VM2's address, the one sitting in the same subnet VM1 can already reach directly.

---
### Step 6: Capture the Routing Table After Adding the Route

On VM1:

```bash
ip route show
```

You should now see a new line:

```
10.0.2.0/24 via 10.0.1.254 dev <private-iface>
```

Save this output as your after evidence. Compare it directly against the before output from Step 4. This single new line is the entire difference, and it is the entire reason connectivity will now work.

---
### Step 7: Verify Connectivity

From VM1:

```bash
ping -c 4 10.0.2.1
```

This should now succeed.

If it does not, work through this checklist in order, since each step rules out one specific cause:

1. Confirm the route actually exists: `ip route show` on VM1, looking for the line from Step 6
2. Confirm IP forwarding is actually enabled on VM2: `sysctl net.ipv4.ip_forward`, must return `1`
3. Confirm a firewall on VM2 is not silently dropping forwarded traffic: `sudo firewall-cmd --list-all`, or temporarily test with `sudo systemctl stop firewalld` to rule firewall out entirely, then turn it back on afterward
4. Confirm VM2 genuinely has both addresses active on its private interface: `ip addr show <private-iface>` on VM2 should list both `10.0.1.254/24` and `10.0.2.1/24`

---
### Step 8: Confirm the Path With Traceroute

From VM1:

```bash
traceroute 10.0.2.1
```

Expected output:

```
traceroute to 10.0.2.1 (10.0.2.1), 30 hops max, 60 byte packets
 1  10.0.1.254 (10.0.1.254)  0.5 ms  0.4 ms  0.4 ms
 2  10.0.2.1 (10.0.2.1)      0.8 ms  0.7 ms  0.7 ms
```

Hop one is VM2, acting as the router. Hop two is the actual destination. This output is direct visual proof that traffic is taking the exact path you configured, through VM2, rather than reaching the destination by some other means.

### Common Issues

**`ip addr add` reports the address is already in use.** Run `ip addr show <private-iface>` first to check for a leftover address from a previous attempt, and remove it with `sudo ip addr del <old-address>/24 dev <private-iface>` before adding the correct one.

**Ping still fails even after confirming the route exists.** This is almost always either IP forwarding still disabled on VM2, or a firewall on VM2 silently dropping forwarded traffic. Work through the Step 7 checklist in the exact order given, since it is specifically ordered from most common cause to least common.

**The two VMs cannot reach each other at all, even within the same subnet.** Confirm both VMs' private interfaces are attached to the same virtual network in your hypervisor settings. In VirtualBox, this typically means both VMs need their second adapter attached to the same Host Only network. If they are attached to two different virtual networks, no amount of correct IP configuration will fix it, since the underlying virtual wiring itself is wrong.
