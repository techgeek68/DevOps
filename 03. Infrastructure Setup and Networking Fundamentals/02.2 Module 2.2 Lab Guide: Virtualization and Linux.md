### Part 1: Verify CPU Virtualization Support

VirtualBox requires Intel VT-x or AMD-V hardware extensions enabled in your BIOS or UEFI.

**On Windows:** Open Task Manager (Ctrl+Shift+Esc), go to the Performance tab, select CPU, and confirm `Virtualization: Enabled`.

**On Linux:**

```bash
grep -E "vmx|svm" /proc/cpuinfo | head -1
```

`vmx` confirms Intel VT-x. `svm` confirms AMD-V. No output means virtualization is either disabled in BIOS or unsupported by your CPU.

**On macOS:**

```bash
sysctl kern.hv_support
```

`kern.hv_support: 1` confirms Apple Hypervisor Framework is active.

If virtualization shows as disabled, reboot into your BIOS or UEFI setup (typically F2, F10, Del, or Esc at startup depending on your manufacturer) and enable Intel Virtualization Technology or AMD-V under the CPU or Advanced tab.

### Part 2: Install VirtualBox

Download the latest version (7.2.x or later) from `virtualbox.org/wiki/Downloads`. Also download the VirtualBox Extension Pack from the same page. It enables USB support and RDP.

Install the main application first, then double click the Extension Pack to install it. Accept the license when prompted.

### Part 3: Download the OS

Choose one of the following:

**CentOS Stream 10:** Download from `centos.org/centos-stream`. Choose the DVD ISO.

> CentOS Stream 10 requires an x86-64-v3 CPU (one supporting AVX2 instructions). If your machine is older, Fedora Server is the safer choice.

### Part 4: Create the Virtual Machine

**Step 1:** Open VirtualBox. Click **New**.

**Step 2:** Configure the machine details:

- Name: `DevOps-BaseVM`
- Folder: A location with sufficient free disk space, create a folder called `DevOps-Lab` on your largest drive
- ISO Image: Leave blank for now, you will attach it after
- Type: **Linux**
- Version: **Red Hat (64-bit)**

**Step 3:** Set hardware resources:

Click **Next**, then **Finish**.

**Step 4:** Configure the network adapter. Right click the VM and select **Settings > Network**. Set Adapter 1 to **Bridged Adapter**. This gives your VM an IP on the same network as your host machine, which makes SSH access straightforward.

**Step 5:** Attach the ISO. Still in **Settings**, go to **Storage**. Click the optical drive that shows **Empty**. Click the disc icon and select **Choose a disk file**. Navigate to your downloaded ISO and select it. Click **OK**.

### Part 5: Install the OS

**Step 2:** Select your language and keyboard layout. Click **Continue**.

**Step 3:** On the Installation Summary screen, configure the required items:

### Part 6: Post Installation Configuration

Log in with the user account you created during installation.

**Update all packages:**

```bash
sudo dnf update -y
```

**Verify network connectivity:**

```bash
ip addr show
ping -c 4 8.8.8.8
```

Both network interfaces should show IP addresses. Ping should succeed.

**Enable and start SSH:**

```bash
sudo systemctl enable --now sshd
sudo systemctl status sshd
```

**Allow SSH through the firewall:**

```bash
sudo firewall-cmd --permanent --add-service=ssh
sudo firewall-cmd --reload
sudo firewall-cmd --list-all
```

**Find the VM's IP address for SSH access:**

```bash
ip addr show
```

Look for the IP on your bridged interface. It will be on the same subnet as your host machine.

**Test SSH from your host machine:**

```bash
ssh <your-username>@<vm-ip-address>
```

Replace both placeholders with your actual values. If this succeeds, your VM is fully accessible over SSH and you no longer need to use the VirtualBox window for most lab work.

**Configure a static IP with nmcli:**

A DHCP address can change after a reboot, making it harder to SSH in consistently. Assign a static IP to the bridged adapter.

First, find your connection name and your current network details:

```bash
nmcli connection show
ip route show | grep default
```

Note the connection name (it might be something like "Wired connection 1") and the default gateway IP.

Then configure the static IP:

```bash
sudo nmcli connection modify "Wired connection 1" \
  ipv4.method manual \
  ipv4.addresses 192.168.1.100/24 \
  ipv4.gateway 192.168.1.1 \
  ipv4.dns "8.8.8.8,8.8.4.4"

sudo nmcli connection down "Wired connection 1"
sudo nmcli connection up "Wired connection 1"
```

Adjust the IP address and gateway to values that make sense for your home network. Verify the change took effect:

```bash
ip addr show
ip route show
```

**Set a permanent hostname:**

```bash
sudo hostnamectl set-hostname devops-base.lab
exec bash
hostnamectl
```

**Take a snapshot:**

In VirtualBox, select **Machine > Take Snapshot**. Name it `Clean Install`. This is your recovery point for the entire course. If a future lab leaves the VM in a broken state, revert to this snapshot and start from a clean baseline.

### Expected Output

- VM boots, installs, and reaches the login prompt without errors
- `ping -c 4 8.8.8.8` succeeds from inside the VM
- `systemctl status sshd` shows `active (running)`
- SSH from your host machine to the VM succeeds
- A static IP is configured and survives a reboot
- A `Clean Install` snapshot exists in VirtualBox

### Common Issues

**VM boots to a black screen or freezes immediately:** Confirm CPU virtualization is enabled in your BIOS. Check that you selected the correct version (Red Hat 64-bit) when creating the VM.

**SSH times out from the host machine:** Confirm `sshd` is running inside the VM with `systemctl status sshd`. Confirm the firewall allows SSH with `firewall-cmd --list-all`. Confirm you are using the correct IP address (run `ip addr show` inside the VM).

**Network interface shows no IP address after boot:** Check that the adapter is set to Bridged Adapter in VirtualBox settings. Toggle the interface connection off and on with `nmcli connection down <name>` and `nmcli connection up <name>`.

### Category 2: File Operations with `grep`, `awk`, `sed`

```bash
# Create a test file to work with

server1 192.168.1.10 web active
server2 192.168.1.11 db active
server3 192.168.1.12 web maintenance

```

**grep: search for patterns**

```bash
# Find lines containing "web"
grep "web" ~/lab-practice/testfile.txt

# Find lines not containing "active" (-v inverts the match)
grep -v "active" ~/lab-practice/testfile.txt

# Find lines containing "db" and count them
grep -c "db" ~/lab-practice/testfile.txt

# Find lines matching a pattern, case insensitive
grep -i "ACTIVE" ~/lab-practice/testfile.txt

# Find lines containing an IP starting with 192.168.1.1
grep "192\.168\.1\.1[0-9]" ~/lab-practice/testfile.txt
```

**awk: process structured text field by field**

```bash
# Print only the first column (server names)
awk '{print $1}' ~/lab-practice/testfile.txt

# Print the first and third columns
awk '{print $1, $3}' ~/lab-practice/testfile.txt

# Print only lines where the fourth column is "active"
awk '$4 == "active" {print $1}' ~/lab-practice/testfile.txt

# Count the number of "web" servers
awk '$3 == "web"' ~/lab-practice/testfile.txt | wc -l

# Print a custom formatted line for each entry
awk '{print "Host:", $1, "| Role:", $3, "| IP:", $2}' ~/lab-practice/testfile.txt
```

**sed: transform text stream by stream**

```bash
# Replace "active" with "online" throughout the file
sed 's/active/online/g' ~/lab-practice/testfile.txt

# Replace only the first occurrence on each line
sed 's/active/online/' ~/lab-practice/testfile.txt

# Delete lines containing "maintenance"
sed '/maintenance/d' ~/lab-practice/testfile.txt

# Print only lines matching a pattern
sed -n '/web/p' ~/lab-practice/testfile.txt

# Edit the file in place (actually modifies the file, not just the output)
sed -i 's/cache/memcache/g' ~/lab-practice/testfile.txt
cat ~/lab-practice/testfile.txt
```

### Category 4: Permissions and Ownership

```bash
# Create files to practice with
touch ~/lab-practice/configs/app.conf
touch ~/lab-practice/scripts/deploy.sh
touch ~/lab-practice/scripts/backup.sh

# View current permissions
ls -la ~/lab-practice/scripts/

# Set permissions using octal notation
chmod 755 ~/lab-practice/scripts/deploy.sh
# 7 = rwx (owner), 5 = r-x (group), 5 = r-x (others)

chmod 600 ~/lab-practice/configs/app.conf
# 6 = rw- (owner), 0 = --- (group), 0 = --- (others)
# A config file containing secrets should be readable only by its owner

# Set permissions using symbolic notation
chmod u+x ~/lab-practice/scripts/backup.sh       # Add execute for the owner
chmod g-w ~/lab-practice/scripts/backup.sh       # Remove write for the group
chmod o-rwx ~/lab-practice/scripts/backup.sh     # Remove all for others

# View the result
ls -la ~/lab-practice/scripts/

# Change ownership
sudo chown root:root ~/lab-practice/configs/app.conf
ls -la ~/lab-practice/configs/

# Change back to your own user
sudo chown $(whoami):$(whoami) ~/lab-practice/configs/app.conf
ls -la ~/lab-practice/configs/

# Change ownership recursively for a directory
sudo chown -R devops-user:devops-team ~/lab-practice/logs
ls -la ~/lab-practice/
```

**Understand the permission bits:**

| Octal | Symbolic | Meaning |
|---|---|---|
| 0 | `---` | No permissions |
| 1 | `--x` | Execute only |
| 4 | `r--` | Read only |
| 5 | `r-x` | Read and execute |
| 6 | `rw-` | Read and write |
| 7 | `rwx` | Read, write, and execute |

## Lab 2.2.C: Bash Mini Project, Health Check Script

### Objective

Write a production style health check shell script from scratch. The script accepts a service name as an argument, checks whether that service is running, logs the result with a timestamp to a log file, and exits with a meaningful exit code. It also checks a list of multiple services in a loop. Schedule it with cron so it runs automatically every five minutes.

### What You Need

- Your Linux VM from Lab 2.2.A with SSH access
- Approximately 60 minutes

### Prerequisites

- Completed Lab 2.2.B, specifically the process management and systemd sections

> **Keep this script.** Lab 2.2.D converts it into a systemd timer. It is also a real artifact you can show in a portfolio.

---

### Part 1: Understand the Requirements

Your script must:

- Accept a service name as a command line argument (`$1`)
- Check whether that service is running using `systemctl is-active`
- Log a timestamped entry to `/var/log/healthcheck.log`
- Exit with code `0` if the service is healthy
- Exit with code `1` if the service is not running
- Define and call at least one function
- Use an `if/else` conditional
- Use a `for` loop to check a list of multiple services when called without arguments

---

### Part 2: Write the Script

Create the script file:

```bash
sudo mkdir -p /opt/scripts
sudo nano /opt/scripts/healthcheck.sh
```

Write the following content. Read every line and understand what it does before moving on:

```bash
#!/bin/bash
# healthcheck.sh
# Checks whether one or more systemd services are running.
# Usage: ./healthcheck.sh [service-name]
# With no argument, checks a default list of services.
# Exit code 0: all checked services healthy
# Exit code 1: one or more services are not running

# -------------------------------------------------------------------
# Configuration
# -------------------------------------------------------------------

LOG_FILE="/var/log/healthcheck.log"
DEFAULT_SERVICES=("sshd" "firewalld" "NetworkManager")

# -------------------------------------------------------------------
# Functions
# -------------------------------------------------------------------

# log_message: writes a timestamped line to the log file and to stdout
log_message() {
    local level="$1"
    local message="$2"
    local timestamp
    timestamp=$(date "+%Y-%m-%d %H:%M:%S")
    echo "[$timestamp] [$level] $message" | tee -a "$LOG_FILE"
}

# check_service: checks one service and returns 0 (healthy) or 1 (unhealthy)
check_service() {
    local service_name="$1"

    if systemctl is-active --quiet "$service_name"; then
        log_message "OK" "Service '$service_name' is running"
        return 0
    else
        log_message "FAIL" "Service '$service_name' is NOT running"
        return 1
    fi
}

# -------------------------------------------------------------------
# Main logic
# -------------------------------------------------------------------

# Ensure the log file exists and is writable
if [ ! -f "$LOG_FILE" ]; then
    sudo touch "$LOG_FILE"
    sudo chmod 644 "$LOG_FILE"
fi

# Track overall health across all checks
overall_exit=0

# If a service name was passed as an argument, check only that one
if [ -n "$1" ]; then
    check_service "$1"
    overall_exit=$?
else
    # No argument given: check the default list in a loop
    log_message "INFO" "Starting health check for default service list"

    for service in "${DEFAULT_SERVICES[@]}"; do
        check_service "$service"
        if [ $? -ne 0 ]; then
            overall_exit=1
        fi
    done

    log_message "INFO" "Health check complete. Exit code: $overall_exit"
fi

exit $overall_exit
```

Save and exit (Ctrl+O then Enter in nano, then Ctrl+X).

### Part 4: Schedule With Cron

Cron is the traditional Unix job scheduler. You schedule jobs by editing a crontab file.

```bash
# Open the root crontab for editing (the script writes to /var/log)
sudo crontab -e
```

This opens the crontab in your default editor. Add this line at the bottom:

```
*/5 * * * * /opt/scripts/healthcheck.sh >> /var/log/healthcheck.log 2>&1
```

Read the cron expression from left to right:

| Field | Value | Meaning |
|---|---|---|
| Minute | `*/5` | Every 5 minutes |
| Hour | `*` | Every hour |
| Day of month | `*` | Every day |
| Month | `*` | Every month |
| Day of week | `*` | Every day of the week |

Save and exit. Verify the crontab was saved:

```bash
sudo crontab -l
```

Wait five minutes, then confirm the log file is being updated:

```bash
tail -20 /var/log/healthcheck.log
```

You should see new entries with timestamps at five minute intervals.

```bash
check_disk_space() {

        return 1
    else
        log_message "OK" "Disk usage on / is at ${usage}%"
        return 0
    fi
}
```

Call it from the main logic block alongside the service checks.

## Lab 2.2.D: systemd Deep Dive

### Objective

Convert the cron based health check from Lab 2.2.C into a proper systemd timer unit. Then verify the timer fires, inspect its output through journald, and understand what systemd gives you that cron does not.

### What You Need

- Your Linux VM with SSH access
- Approximately 60 minutes

### Prerequisites

- Read Section 2.2 of this module, specifically the systemd section
- Lab 2.2.C completed and tested

---

### Part 1: Understand What You Are Building

A systemd timer requires two paired unit files: a `.timer` file that defines the schedule, and a `.service` file that defines what to run when the timer fires. Together they replace the cron entry from Lab 2.2.C.

Systemd timers give you several things cron does not:

- The job output goes into the journal automatically, readable with `journalctl -u`
- The timer can depend on other systemd units (only fire after network is up, for example)
- You can inspect the timer's schedule and next fire time with `systemctl status`
- If the system was off when a timer was supposed to fire, `Persistent=true` makes it fire immediately when the system comes back online
- `systemctl list-timers` gives you a clear view of all scheduled jobs in one place

---

### Part 2: Write the Service Unit File

Create the service unit file:

```bash
sudo nano /etc/systemd/system/healthcheck.service
```

Enter the following content:

```ini
[Unit]
Description=System Health Check Script
Documentation=man:systemd.service(5)
After=network.target

[Service]
Type=oneshot
ExecStart=/opt/scripts/healthcheck.sh
User=root
StandardOutput=journal
StandardError=journal
SyslogIdentifier=healthcheck
```

`Type=oneshot` is the correct type for a script that runs, completes, and exits, as opposed to a long running daemon that stays alive. systemd considers a oneshot service complete when the `ExecStart` process exits, regardless of the exit code.

`StandardOutput=journal` and `StandardError=journal` direct both the script's regular output and any error output into the journal, where you can query them with `journalctl`. Without these, output would be lost.

`SyslogIdentifier=healthcheck` gives every journal entry a clean identifier you can filter on: `journalctl -u healthcheck.service` or `journalctl -t healthcheck`.

### Part 3: Write the Timer Unit File

Create the timer unit file with the same base name as the service:

```bash
sudo nano /etc/systemd/system/healthcheck.timer
```

Enter the following:

```ini
[Unit]
Description=Run System Health Check Every 5 Minutes
Documentation=man:systemd.timer(5)

[Timer]
OnBootSec=2min
OnUnitActiveSec=5min
Persistent=true
Unit=healthcheck.service

[Install]
WantedBy=timers.target
```

`OnBootSec=2min` fires the timer two minutes after boot. Without this, a timer using only `OnUnitActiveSec` would not fire at all until its first manual trigger.

`OnUnitActiveSec=5min` fires the timer five minutes after the last time the associated service activated. Combined with `OnBootSec`, this produces a run two minutes after boot, then every five minutes thereafter.

`Persistent=true` means if the system was off when the timer was supposed to fire, it will fire immediately the next time the system boots. For a health check, this ensures no check is ever permanently skipped due to downtime.

### Part 4: Enable and Start the Timer

Tell systemd about the new unit files:

```bash
sudo systemctl daemon-reload
```

Enable the timer so it starts automatically at boot:

```bash
sudo systemctl enable healthcheck.timer
```

Start it now without waiting for a reboot:

```bash
sudo systemctl start healthcheck.timer
```

Verify the timer is running:

```bash
sudo systemctl status healthcheck.timer
```

Expected output includes `Active: active (waiting)` and shows the next trigger time.

List all active timers to confirm yours appears:

```bash
sudo systemctl list-timers
```

Your timer should appear in the list with the `NEXT` column showing when it will next fire.

### Part 7: Override a Unit File Without Editing It Directly

Unit files in `/etc/systemd/system/` are yours to edit directly. But for files in `/lib/systemd/system/` that came with packages, editing directly risks being overwritten on the next package update. Practice the correct override approach:

```bash
# Create an override for your healthcheck service
sudo systemctl edit healthcheck.service
```

This opens an empty override file. Add a change, for example a description update or an extra environment variable:

```ini
[Service]
Environment=CHECK_ENV=lab
```

Save and exit. The override file is saved to `/etc/systemd/system/healthcheck.service.d/override.conf`. View the merged result:

```bash
sudo systemctl cat healthcheck.service
```

You will see the base file and the override block merged together, with the override clearly marked. Reload and restart to apply:

```bash
sudo systemctl daemon-reload
sudo systemctl restart healthcheck.service
sudo journalctl -u healthcheck.service -n 5
```

### Common Issues

**`daemon-reload` required but forgotten:** If the timer shows stale information or a unit change does not take effect, always run `sudo systemctl daemon-reload` before starting or restarting the unit. This is the most common systemd mistake in practice.

**Timer shows `Active: inactive (dead)` instead of `Active: active (waiting)`:** Confirm you ran `sudo systemctl start healthcheck.timer`, not just `enable`. Enable adds it to the boot sequence. Start makes it run right now.

**`journalctl -u healthcheck.service` shows no entries:** Confirm the service was actually triggered. Run `sudo systemctl start healthcheck.service` manually and immediately run the journalctl command again. If the service shows as `failed`, run `systemctl status healthcheck.service` and read the error in the Status line.

**Script works from the command line but fails when run by systemd:** systemd runs services in a minimal environment without your shell's environment variables. If your script depends on `PATH` or other variables set in your `.bashrc`, define them explicitly in the unit file with `Environment=` or `EnvironmentFile=`.
