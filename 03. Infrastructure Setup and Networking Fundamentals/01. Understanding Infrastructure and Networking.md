## FM2.1 What Is Virtualization

### The Problem Before Virtualization

Imagine a company in the early 2000s. They need a web server, a database server, an email server, and a file server. The standard practice at the time was to buy four physical machines and run one workload on each.

This created a real problem. Most servers spend the majority of their time doing almost nothing. A file server checked twice a day uses a tiny fraction of its CPU and memory. The other 95 percent of that machine's capacity sits there unused, while the company has still paid for the hardware, the electricity, the cooling, and the physical rack space.

On top of that, every physical machine is a separate thing to maintain, patch, and worry about. If a company has 40 different applications, the traditional model meant 40 different machines.

---
### What Virtualization Actually Does

Virtualization lets a single physical computer pretend to be several separate computers at once. Each of these separate, software-based computers is called a virtual machine, usually shortened to VM. Each VM has its own operating system, its own allocated memory, its own virtual disk, and behaves exactly like a real, standalone computer from the perspective of anything running inside it.

The physical machine underneath is called the host. The VMs running on it are called guests.

A simple way to picture this: imagine a single large building divided into several fully separate apartments. Each apartment has its own kitchen, its own door, its own electricity meter. The tenants in one apartment have no idea what is happening in another. The building is the host. Each apartment is a VM.

---
### Why This Matters Day to Day

Once you can run several virtual machines on one physical computer, several things become possible that were previously expensive or impossible.

You can run a different operating system on each VM. One VM can run a Linux based web server, another can run a Windows application, all on the very same physical hardware.

You can isolate problems. If one VM crashes or gets misconfigured, the other VMs on the same host keep running normally, because each VM is sealed off from the others.

You can copy a VM. Once you have a working virtual machine set up exactly the way you want it, you can duplicate it in minutes instead of setting up a new physical machine from scratch, which used to take hours or days.

You can undo mistakes instantly. Most virtualization software lets you take a snapshot, a saved copy of the VM at a specific point in time. If you break something while experimenting, you simply roll back to the snapshot instead of starting over.

This is exactly why this entire course is built around virtual machines. You are going to build a whole lab environment, a web server, an application server, several Kubernetes nodes, without buying a single piece of additional hardware. All of it will run as virtual machines on your one laptop or desktop.

## FM2.2 Hypervisors: The Software That Makes Virtualization Possible
---
### What a Hypervisor Is

A hypervisor is the piece of software responsible for creating virtual machines and managing how they share the physical hardware underneath them. It decides how much CPU time each VM gets, how much memory each VM is allowed to use, and it keeps every VM isolated from every other VM.

There are two fundamentally different ways a hypervisor can be set up, and the difference matters quite a lot.

---
### Bare Metal Hypervisors (Hypervisor Type 1)

A bare metal hypervisor installs directly onto the physical hardware. There is no separate operating system underneath it. The hypervisor itself essentially becomes the operating system of that machine, and every VM runs directly on top of it.

Because there is no extra software layer in between the hardware and the VM, bare metal hypervisors are fast and efficient. This is the setup you will find in real data centers and in the infrastructure behind every major cloud provider. When you launch a virtual machine on AWS, it is running on a bare metal hypervisor somewhere in an AWS data center.

Common examples include VMware ESXi, Microsoft Hyper-V, and KVM, which is built directly into the Linux kernel itself.

---
### Hosted Hypervisors (Hypervisor Type 2)

A hosted hypervisor is different. It is installed as a regular application on top of an operating system you already have running, the same way you would install a web browser or a text editor. Your laptop still boots into Windows, macOS, or Linux as normal, and the hypervisor runs as a program inside that operating system.

This is less efficient than a bare metal hypervisor because every request a VM makes to the hardware has to pass through your regular operating system first. But it is far easier to set up. You do not need a dedicated server. You can install it on the same laptop you already use for everything else.

This is exactly the kind of hypervisor used throughout this course. VirtualBox and VMware Workstation are both hosted hypervisors, and they are what you will use to build your entire lab environment on your own machine.

---
### Why the Distinction Matters for You

You will be using a hosted hypervisor for every lab in this course because you almost certainly do not have a spare dedicated server sitting around. But it is important to understand that the production servers you eventually deploy real applications to, the actual web servers and Kubernetes nodes running in a company's infrastructure, are running on bare metal hypervisors. The skills transfer directly. A virtual machine behaves the same way to you whether the hypervisor underneath it is bare metal or hosted. The difference is about performance and where the hypervisor sits, not about how you use the VM.

## FM2.3 What a Network Actually Is

### The Basic Idea

A network is simply a group of devices that are able to send information to each other. That is the entire concept at its core. Your laptop, your phone, the server running this course's lab environment, and every website you have ever visited are all part of one enormous interconnected network that we call the internet.

But before any of that complexity, you need to understand the much smaller building blocks that make it work: addresses and the rules for how devices find and talk to each other.

---
### Why You Need an Address

Every device on a network needs a unique address so other devices know exactly where to send information. This is conceptually identical to a postal address. If you want to mail a letter to a specific house, you need that house's exact street address, not just the name of the city.

On a network, this address is called an IP address, where IP stands for Internet Protocol. An IP address looks like this: `192.168.1.10`. It is a unique label assigned to a device so that other devices on the network know exactly where to deliver information meant for it.

---
### IPv4 Addresses in Plain Terms

The most common type of IP address you will work with is called IPv4. It is written as four numbers separated by dots, like `192.168.1.10`. Each of those four numbers can range from 0 to 255.

You do not need to memorize the math behind why the range stops at 255 right now. What matters at this stage is recognizing the shape of an IP address on sight and understanding that it identifies one specific device on a network.

---
### Public Addresses vs Private Addresses

Not every IP address is reachable from anywhere on the internet. There is an important distinction between the two kinds of addresses.

A public IP address is globally unique and can be reached from anywhere on the internet. Your home router has a public IP address that anyone on the internet could technically attempt to contact.

A private IP address is only valid inside a local network, like your home Wi-Fi or a company's internal office network. These addresses are not reachable directly from the internet. Your laptop, your phone, and your smart TV at home all have private IP addresses inside your home network, usually starting with `192.168.` something. There are two other private ranges you will see constantly once you start working with larger networks and cloud infrastructure: addresses starting with `10.` and addresses in the `172.16.` through `172.31.` range. All three ranges are private by definition, no matter which network they appear on.

This is why two completely different households can both have a device at `192.168.1.10` without any conflict. Those addresses only have meaning inside their own private network. Out on the public internet, they mean nothing.

This distinction becomes very important later when you start working with cloud infrastructure, because cloud providers build entire private networks for you to place your servers inside, and only specific parts of that private network are made reachable from the public internet.

---
### What a Subnet Is, In Plain Terms

A subnet, short for subnetwork, is a smaller network carved out of a larger one. Instead of having every single device in a company on one giant, flat network, engineers divide the address space into smaller, organized chunks.

A simple analogy: imagine a large apartment complex with hundreds of units. Instead of giving every single resident a random unconnected address, the complex is organized into buildings, and each building has its own clearly defined range of unit numbers. A subnet is like one of those buildings: a smaller, defined chunk of addresses carved out of the larger complex.

Why bother dividing things up like this? Three main reasons. It keeps networks organized, so engineers know exactly which group of devices lives where. It limits how far network problems can spread, since issues in one subnet do not automatically affect devices in another. And it allows different groups of machines, like web servers, databases, and internal admin tools, to be kept separate from each other for both organizational and security reasons.

You will work through the actual math of subnetting in Module 2's labs. For now, the concept to hold onto is simple: a subnet is a defined, organized slice of a larger address space, and dividing a network into subnets is standard practice in any real infrastructure.

## FM2.4 How Your Computer Finds Things by Name
---
### The Problem DNS Solves

Nobody wants to memorize a string of numbers to visit a website. You do not type something like `140.82.121.4` into your browser to reach GitHub (the exact number varies and can change). You type `github.com`, and somehow your computer figures out the rest.

The system responsible for this translation is called DNS, which stands for Domain Name System. Its entire job is to take a name that a human can remember, like `github.com`, and translate it into the IP address that computers actually need in order to connect to the right machine.

---
### A Simple Way to Picture It

Think of DNS as a giant, distributed phone book for the internet. When you look up a friend's name in a phone book, you find their phone number. DNS does the same thing, except it looks up a website's name and returns its IP address. The crucial difference is that this phone book is not stored in one single place. It is spread across thousands of servers around the entire world, each one responsible for a different piece of the puzzle.

---
### What Happens When You Type a Website Name

When you type a domain name into your browser, your computer does not already know the answer. It asks a question, and that question gets passed along a chain of different servers until someone has the answer.

First, your computer checks if it already knows the answer from a recent lookup. If it does, it uses that immediately, since asking again would be a waste of time.

If it does not already know, it asks a resolver, a server whose job is to go find the answer on your behalf. This is often run by your internet provider, or you might be configured to use a public one like Google's resolver.

That resolver then works through the distributed phone book system, eventually finding the server that holds the authoritative answer for that specific domain name, and that answer makes its way back to your computer.

The entire process usually takes well under a second, even though it might involve several servers scattered across different parts of the world.

---
### Why This Matters for You

As you move further into this course, you will work with DNS directly: configuring it, inspecting it, and occasionally troubleshooting it when something does not resolve the way you expect. Understanding this basic chain of lookups now will make every one of those later exercises make sense immediately, instead of feeling like a mysterious black box.

## FM2.5 What a Firewall Does
---
### The Basic Idea

A firewall is a piece of software, or sometimes dedicated hardware, that controls what network traffic is allowed in and out of a device or a network. Think of it as a security guard standing at the only entrance to a building, checking everyone who tries to come in or go out against a list of rules.

Without a firewall, any device on the internet could attempt to connect to any open service running on your server. A firewall lets you decide exactly what is allowed through and what gets blocked.

---
### Why Every Server Needs One

Imagine you set up a server and forget about a firewall entirely. That server might be running several different services in the background, perhaps a database, a remote login service, and a few other things you installed for testing and forgot to remove. Without a firewall, every single one of those services is potentially reachable by anyone on the internet who knows or guesses the right address and port.

A properly configured firewall only allows the specific traffic you actually intend to allow. If your server is meant to be a web server, the firewall would typically allow traffic on the standard web ports and block essentially everything else. This dramatically shrinks the number of ways someone could try to break into that machine.

---
### Inbound and Outbound Traffic

Firewalls generally think about traffic in two directions.

Inbound traffic is anything trying to come into your server from somewhere else. A request from a browser somewhere on the internet trying to load a webpage on your server is inbound traffic.

Outbound traffic is anything leaving your server, heading out to somewhere else. Your server reaching out to download a software update is outbound traffic.

A firewall can be configured to control both directions independently. A common, sensible default many engineers use is to be quite strict about what is allowed in, while being more permissive about what is allowed out, since a server initiating its own outbound connections is generally less risky than an unknown party initiating an inbound one.

You will configure real firewall rules yourself in Module 2's labs. The concept to internalize here is simply that a firewall is a gatekeeper, and that gatekeeper needs a clear, deliberate set of rules rather than being left wide open or, just as commonly, accidentally configured to block traffic you actually needed to allow.

## FM2.6 Why a Single Server Is Never Enough
---
### The Problem at Scale

Imagine a single web server handling all the traffic for a popular website. For a while, this works fine. But as more and more people start visiting that site at the same time, that one server eventually reaches its limit. It runs out of capacity to handle more requests, and visitors start experiencing slow load times or, worse, the site goes down entirely.

There is also a second, equally serious problem. If that one server fails for any reason, a hardware fault, a software crash, anything at all, the entire website goes offline. There is nothing left to take over and keep serving visitors.

---
### What a Load Balancer Does

A load balancer is a piece of software, or sometimes dedicated hardware, that sits in front of several servers and spreads incoming traffic across all of them, rather than letting it all pile onto one single machine.

A useful analogy is a grocery store with several checkout lanes open at once, with someone directing each new customer to whichever lane is shortest. The load balancer plays exactly that role for network traffic, directing each new incoming request to whichever server is best positioned to handle it.

This solves both problems at once. Traffic gets spread out, so no single server gets overwhelmed on its own. And if one server in the group fails, the load balancer simply stops sending traffic to it and continues distributing requests among the remaining healthy servers. From the perspective of someone visiting the website, nothing appears to have gone wrong at all.

---
### Why This Matters for You

You will work extensively with load balancing later in this course, configuring it directly and exploring several different strategies for how traffic gets distributed. For now, the concept to hold onto is straightforward: any serious, real world application is rarely running on just one server, and the load balancer is the component responsible for making a whole group of servers behave, from the outside, like one single, dependable, always available service.
