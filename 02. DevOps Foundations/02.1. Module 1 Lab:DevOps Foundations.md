### Background: Why Approximate?

DORA metrics in production are measured through CI/CD platforms, deployment logs, and incident management systems. For open source projects, you do not have access to their internal tooling. Instead, you will use GitHub's public data as proxies:

| DORA Metric | Proxy You Will Use |
|---|---|
| Deployment Frequency | Number of GitHub Releases per time window |
| Lead Time for Changes | Time from PR open to merge, plus time from merge to release |
| Change Failure Rate | Patch releases published within 48 hours of a minor or major release |
| MTTR | Time from bug issue opened to fix released |

These proxies are approximations. In your report, you will acknowledge where each proxy is imprecise and what a more accurate measurement would require.

### Step 2: Measure Deployment Frequency

1. Open the repository on GitHub and click **Releases** in the right sidebar.
2. Count the number of releases published in the last **90 days**. Include all types: major, minor, and patch.
3. Calculate the average release frequency:

```
releases in 90 days / 90 = average releases per day
```

4. If releases per day is less than 1, convert it:

```
90 / releases in 90 days = average days between releases
```

Record the raw count and your calculated frequency.

**Classify your result:**

| Frequency | DORA Level |
|---|---|
| Multiple per day | Elite |
| Between once per day and once per week | High |
| Between once per week and once per month | Medium |
| Less than once per month | Low |

---
> **What to pay attention to:** Many projects release patch versions frequently, but minor and major versions rarely. If you see this pattern, note it. It tells you the team can respond quickly to bugs, but major feature work takes longer to reach users.

### Step 3: Measure Lead Time for Changes

Lead time measures how long code takes to travel from a developer's commit to a production deployment.

1. Go to the **Pull Requests** tab. Filter by **Closed**. Sort by **Newest**.
2. Select five recently merged PRs. Choose feature or bug fix work. Skip documentation-only PRs.
3. For each PR, collect:
   - Date the PR was opened (visible in the PR header)
   - Date the PR was merged (visible in the merge event at the bottom)
   - The next release published after this merge (look at the Releases page for the first release after the merge date)
   - Date of that release

4. For each PR, calculate:

```
Part A: PR open to merge = merge date minus open date (in days)
Part B: Merge to release = release date minus merge date (in days)
Lead time approximation = Part A + Part B
```

5. Average the five lead time values.

Create a table like this for your five PRs:

| PR Title | Opened | Merged | Part A (days) | Released | Part B (days) | Total Lead Time |
|---|---|---|---|---|---|---|
| | | | | | | |
| | | | | | | |
| | | | | | | |
| | | | | | | |
| | | | | | | |
| **Average** | | | | | | |

**Classify your average:**

| Lead Time | DORA Level |
|---|---|
| Less than 1 hour | Elite |
| 1 day to 1 week | High |
| 1 week to 1 month | Medium |
| More than 1 month | Low |

**Note on open-source projects:** PR review cycles in open-source projects are often longer than in commercial teams because reviewers are volunteers. If lead time is 2 to 3 weeks, this does not necessarily mean the project is poorly run. Note this nuance in your report.

### Step 4: Measure Change Failure Rate

Change failure rate measures what percentage of deployments cause a problem requiring remediation.

In open source projects, emergency patch releases published shortly after a minor or major release are the best proxy for this.

1. List the last **10 releases** for your repository in a table:

| Release Tag | Published Date | Type (major/minor/patch) | Followed by emergency patch? |
|---|---|---|---|
| | | | |

2. For each non-patch release (e.g., v2.4.0), check whether a patch release (e.g., v2.4.1) was published within **72 hours**.

3. Calculate:

```
emergency patches / total releases examined * 100 = approximate change failure rate (%)
```

**Classify your result:**

| Rate | DORA Level |
|---|---|
| 0% to 15% | Elite or High |
| 16% to 30% | Medium |
| More than 30% | Low |

**If the rate is 0%:** This could mean excellent quality control, or it could mean the project does not tag emergency patches separately (they may ship fixes in the next regular release). Investigate one or two releases in detail and note your finding.

### Common Issues

**The Releases tab is empty:** Check the repository README for alternative release channels. Some projects publish to npm, PyPI, DockerHub, or use GitHub Packages. If you genuinely cannot find release data, pick a different repository from the recommended list.

**PR lead times are extremely long (months):** This is common in large open-source projects where reviewers are maintainers with day jobs. Your report should note this explicitly as a structural difference from commercial teams and explain how this affects your lead time measurement.

**Cannot find bug labeled issues:** Try searching Issues for the terms "regression", "panic", "crash", or "fix". Many projects use custom labels. If you cannot find five qualifying issues, use three and note the limitation.

**Git push fails:** Make sure the remote repository exists on GitHub and that your local `origin` remote URL is correct. Run `git remote -v` to verify.

### Common Mistakes

**Setting SLO and SLA to the same number:** The SLA must always be lower than the SLO. If the SLO is breached, you need room to recover before the SLA is breached. If they are the same, every SLO breach is immediately a contractual violation.

**Not showing calculations:** In a real engineering environment, your SLO decisions need to be defensible with data. Get into the habit of showing the math.

**Making the SLO too aspirational:** An SLO that you cannot actually meet is not an SLO. It is a wish. Start from what you can measure, set the target slightly below your current best performance, and tighten it over time.

**Forgetting the time window:** "99.9% availability" is meaningless without a time window. The same outage consumes a tiny fraction of the yearly budget but more than half of the daily budget. Always specify the window.

### Step 2: Write the Terraform Configuration

```bash

  region = "us-east-1"

    from_port   = 22
    to_port     = 22
    protocol    = "tcp"

  }

    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]

  }

  tags = {
    Name        = "devops-lab-drift-sg"
    Environment = "lab"
    ManagedBy   = "terraform"
    Purpose     = "drift-detection-exercise"
  }

output "security_group_id" {
```

### Step 5: Introduce Drift

This simulates a common real-world scenario: an engineer makes a quick manual change directly in AWS during an incident and never updates the Terraform code.

```bash
# Capture the security group ID into a variable
SG_ID=$(terraform output -raw security_group_id)
echo "Working with security group: $SG_ID"

# Add an HTTP rule directly through the AWS CLI, bypassing Terraform entirely
aws ec2 authorize-security-group-ingress \
  --group-id "$SG_ID" \
  --protocol tcp \
  --port 80 \
  --cidr "0.0.0.0/0"

# Add an HTTPS rule the same way
aws ec2 authorize-security-group-ingress \
  --group-id "$SG_ID" \
  --protocol tcp \
  --port 443 \
  --cidr "0.0.0.0/0"

echo "Two manual rules added. Drift has been introduced."
```

Verify the drift is now real in AWS:

```bash
aws ec2 describe-security-groups \
  --filters "Name=tag:Name,Values=devops-lab-drift-sg" \
  --query "SecurityGroups[*].IpPermissions[*].{Port:FromPort,Protocol:IpProtocol}" \
  --output table
```

You should see three rules now: SSH (22), HTTP (80), and HTTPS (443).

---
### Step 6: Detect the Drift

Run `terraform plan`. Terraform will:
1. Read your `main.tf` (desired state: only port 22)
2. Read the state file (recorded state: only port 22)
3. Query AWS (actual state: ports 22, 80, and 443)
4. Compute the diff between desired and actual

```bash
terraform plan
```

Read the output carefully. Pay attention to the `-` and `~` symbols.

**What you should see:**

```
  ~ update in-place

  ~ resource "aws_security_group" "lab_sg" {
        id   = "sg-0xxxxxxxxxxxxxxxxx"
        name = "devops-lab-drift-sg"

      ~ ingress = [
          - {
              - cidr_blocks      = ["0.0.0.0/0"]
              - description      = ""
              - from_port        = 443
              - protocol         = "tcp"
              - to_port          = 443
            },
          - {
              - cidr_blocks      = ["0.0.0.0/0"]
              - description      = ""
              - from_port        = 80
              - protocol         = "tcp"
              - to_port          = 80
            },
            {
                cidr_blocks      = ["0.0.0.0/0"]
                description      = "SSH access"
                from_port        = 22
                protocol         = "tcp"
                to_port          = 22
            },
        ]
    }

Plan: 0 to add, 1 to change, 0 to destroy.
```

The `-` prefix means Terraform will remove those rules on the next apply. They are not in the code, so Terraform does not consider them authoritative.

**This is the critical point:** Terraform treats your code as the source of truth, not the current state of AWS. A manual change made directly in AWS is not legitimate in Terraform's model. It will be removed.

---
### Step 7: Enforce Declared State

Apply to eliminate the drift:

```bash
terraform apply
# Type 'yes'
```

Verify:

```bash
aws ec2 describe-security-groups \
  --filters "Name=tag:Name,Values=devops-lab-drift-sg" \
  --query "SecurityGroups[*].IpPermissions[*].{Port:FromPort,Protocol:IpProtocol}" \
  --output table
```

You should see only one rule: port 22. The HTTP and HTTPS rules are gone.

---
### Step 8: Add the Rules Correctly

Now add HTTP and HTTPS the right way, through the Terraform code.

Open `main.tf` and add two new ingress blocks inside the `aws_security_group` resource, after the existing SSH block:

```hcl
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTP access"
  }

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTPS access"
  }
```

Apply the change:

```bash
terraform plan
# Verify the plan shows adding exactly two ingress rules

terraform apply
```

Verify all three rules are now present:

```bash
aws ec2 describe-security-groups \
  --filters "Name=tag:Name,Values=devops-lab-drift-sg" \
  --query "SecurityGroups[*].IpPermissions[*].{Port:FromPort,Protocol:IpProtocol}" \
  --output table
```

The key difference from Step 5: the rules are now in your code, in Git, reviewable, auditable, and reproducible. They will survive the next `terraform apply` instead of being removed.

### Troubleshooting

**`terraform init` fails with provider error:**
Run `terraform version` first. If Terraform version is below 1.5, upgrade it. If behind a corporate proxy, set `HTTPS_PROXY=http://your-proxy:port` before running `terraform init`.

**`aws configure` credentials not working:**
Run `aws sts get-caller-identity`. If this returns an error, your access key is wrong or expired. Generate a new access key from the AWS IAM console under your user's Security Credentials tab.

**Security group name already exists:**
If you ran this lab before without destroying, the group still exists. Run:
```bash
SG_ID=$(aws ec2 describe-security-groups \
  --filters "Name=group-name,Values=devops-lab-drift-sg" \
  --query "SecurityGroups[0].GroupId" --output text)
terraform import aws_security_group.lab_sg $SG_ID
terraform destroy
```

**`terraform plan` shows "No changes" after Step 5:**
Terraform's local state may be stale. Force a refresh:
```bash
terraform apply -refresh-only
terraform plan
```
