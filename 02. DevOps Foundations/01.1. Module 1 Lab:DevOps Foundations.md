## Lab 1.A DORA Baseline: Measuring a Real Repository

### Objective

Use a real open source GitHub repository to approximate DORA metrics from publicly available data. This exercise builds the habit of reading delivery signals from engineering activity, which is what you will do when assessing the health of a team or pipeline in a real job.

### What You Need

- A GitHub account (free)
- Access to the GitHub web UI (no terminal required for this lab)
- Approximately 60 minutes

### Prerequisites

Read Section 1.1, specifically the DORA Four Key Metrics section, before starting. You need to understand what each metric measures before you can find it in real data.

### Background

DORA metrics are normally measured through your CI/CD platform and deployment tooling. For this lab, you will approximate them using GitHub's public data: commit history, release tags, pull request timelines, and issue trackers. The approximations will not be perfect, but they will be close enough to be instructive.

---
### Step 1: Choose a Repository

Pick a well maintained open source project with an active release history. Good candidates:

- `github.com/grafana/grafana`
- `github.com/kubernetes/kubernetes`
- `github.com/hashicorp/terraform`
- `github.com/prometheus/prometheus`
- Any active project you already use or are curious about

>Avoid repositories that have not had a release in the last six months. You need recent activity.

---
### Step 2: Measure Deployment Frequency

In open source projects, a GitHub Release corresponds to a production deployment.

1. Go to your chosen repository on GitHub.
2. Click **Releases** in the right sidebar.
3. Count the number of releases published in the last 90 days.
4. Calculate: `releases per 90 days / 90 = releases per day`

Record your result and classify it using the DORA performance levels:

| Your Result | DORA Level |
|---|---|
| Multiple per day | Elite |
| Once per day to once per week | High |
| Once per week to once per month | Medium |
| Less than once per month | Low |

**What to look for:** Some projects release minor versions frequently and major versions rarely. Note both. A project releasing patch versions daily but minor versions monthly has a nuanced story: the team can ship, but major feature work takes longer to reach users.

---
### Step 3: Approximate Lead Time for Changes

Lead time is measured from when code is committed to when it reaches production. For an open-source project, you can approximate this using pull request data.

1. Go to the **Pull Requests** tab and filter by **Closed**.
2. Pick five recently merged PRs that represent feature or bug fix work (skip documentation-only PRs).
3. For each PR, record:
   - The date the PR was opened
   - The date it was merged
   - Whether it was included in a release (check if a release was published within a few days after the merge)
4. Calculate the average time from PR open to release publication.

This is not the exact lead time (lead time starts at commit, not PR open), but it is a reasonable approximation for this exercise.

Record your average and classify it using the DORA performance levels:

| Your Result | DORA Level |
|---|---|
| Less than 1 hour | Elite |
| 1 day to 1 week | High |
| 1 week to 1 month | Medium |
| More than 1 month | Low |

---
### Step 4: Approximate Change Failure Rate

Change failure rate requires knowing how many deployments caused a problem. In open source repositories, hotfix releases and patch releases that follow immediately after a minor or major release are a proxy for this.

1. Look at the last 10 releases in the repository.
2. Count how many were followed by a patch release within 48 hours (e.g., v1.4.0 followed by v1.4.1 two days later).
3. Calculate: `emergency patches / total releases * 100 = approximate change failure rate`

**Note:** This is a rough proxy. In a real environment, you would measure this using your incident management system correlated with your deployment log.

---
### Step 5: Approximate MTTR

Mean Time to Recovery in an open source project can be approximated using GitHub Issues labeled as bugs or incidents.

1. Go to **Issues** and filter by label. Look for labels like `bug`, `critical`, `incident`, or `regression`.
2. Pick five closed issues that represent bugs or service degradations.
3. For each issue, record:
   - Date the issue was opened
   - Date the issue was closed (a fix was merged and released)
4. Calculate the average time from issue open to close.

This approximates MTTR for a software bug (time from detection to fix shipped). It does not capture infrastructure incidents, but it gives you a signal about how quickly the team resolves quality problems.

---
### Step 6: Write Your Analysis

Write a short analysis (approximately 300 words) covering:

1. Which repository did you choose and why?
2. Your approximate values for all four DORA metrics.
3. The DORA performance level for each metric (elite, high, medium, low).
4. One specific observation: did any metric surprise you? Was there a metric where the data was hard to find or the proxy felt unreliable?
5. If this were your team's repository, which one metric would you focus on improving first, and what specific engineering change would you make?

### Expected Output

- A table with your four DORA metric approximations and DORA levels
- Your 300 word analysis

---
### Common Issues

**Cannot find release data:** Some repositories use a different release mechanism (npm publish, Docker Hub pushes, or GitHub Actions artifacts). If the Releases tab is empty, check the repository's README for how they publish versions.

**PR timelines seem very long:** Some open-source projects have very long PR review cycles because reviewers are volunteers and not always available. This is different from a commercial team. Note this in your analysis.

**Change failure rate is zero:** For a large, well-maintained project this is plausible. Note it and consider whether the absence of emergency patches reflects genuine quality or whether the project simply does not tag emergency patches separately.

## Lab 1.B: SLO and Error Budget Calculation

### Objective

Work through the mathematics of SLIs, SLOs, and error budgets so that when you are asked to define reliability targets for a real service, you can do it correctly without guessing.

### What You Need

- A calculator or spreadsheet (Google Sheets or Excel)
- Approximately 45 minutes
- No terminal required

### Prerequisites

Read the SRE vs. DevOps section of Section 1.1, specifically the SLI, SLO, SLA, and error budget subsections.

---

### Part 1: Calculate Error Budgets

For each of the following SLOs, calculate the error budget in minutes per 30-day month.

A 30-day month contains exactly 43,200 minutes.

| SLO | Error Budget (minutes per 30 days) |
|---|---|
| 99% availability | ? |
| 99.5% availability | ? |
| 99.9% availability | ? |
| 99.95% availability | ? |
| 99.99% availability | ? |

**Formula:** `(1 - SLO as decimal) * 43,200 = error budget in minutes`

**Example:** For 99.9%: `(1 - 0.999) * 43,200 = 0.001 * 43,200 = 43.2 minutes`

Fill in the table. Notice what happens to the error budget as you move from 99% to 99.99%. The difference between 99.9% and 99.99% is only 0.09 percentage points but it cuts the error budget from 43 minutes to 4 minutes. This is why SLO targets are chosen carefully.

---

### Part 2: Scenario Analysis

Work through each scenario and answer the questions.

**Scenario A: The Payment Service**

Your payment service has an SLO of 99.95% availability over a rolling 30-day window. It is day 20 of the current window.

On day 18, a deployment caused a 6-minute outage.
On day 20, a database failover caused a 3-minute degradation.

Questions:

1. What is the total error budget for the 30-day window in minutes?
2. How many minutes of error budget have been consumed so far?
3. How many minutes remain for the rest of the window?
4. Two planned database maintenance windows are scheduled for days 22 and 27, each expected to take approximately 4 minutes. Can both proceed without breaching the SLO? Show your calculation.
5. If both maintenance windows proceed and one of them runs long by 2 minutes (6 minutes instead of 4), what is the impact?

**Scenario B: The Search API**

Your search API has a latency SLO: 95% of requests must complete in under 200ms, measured over a rolling 7-day window.

A 7-day window contains 10,080 minutes. At peak load the service handles approximately 500 requests per minute.

Questions:

1. Total requests over a 7-day window at 500 req/min: `500 * 10,080 = ?`
2. At 95% SLO, how many requests are allowed to exceed 200ms?
3. On day 5, you receive a report that the 95th percentile latency has risen to 350ms for the past 2 hours. Assuming the same request rate, how many requests in those 2 hours exceeded your 200ms SLO threshold?
4. Is this a breach of the SLO for the full 7-day window? Show why or why not.

**Scenario C: Setting the SLA**

Your engineering team has agreed to an internal SLO of 99.9% availability. The business team wants to publish an SLA to enterprise customers.

Questions:

1. What availability percentage would you recommend for the external SLA? Explain your reasoning.
2. If your SLO is breached (you drop below 99.9%) but your SLA is not, what does that tell you about your buffer?
3. A customer asks for a 99.95% SLA. What would you need to change on the engineering side before agreeing to this?

---

### Part 3: Design Your Own SLI and SLO

Choose one of the following services:

- A user authentication API (handles login and token validation)
- A file upload service (accepts files up to 100MB from users)
- A notification delivery service (sends emails and SMS messages)

For your chosen service, define:

1. **Three SLIs:** For each one, specify the exact measurement (numerator, denominator, time window, and how you would collect it).
2. **Three SLOs:** One for each SLI. Set targets that are achievable but meaningful. Do not set 100%.
3. **Error budget:** Calculate the error budget in minutes per 30 days for your most critical SLO.
4. **One SLA:** Propose an external SLA percentage based on your SLO, and explain the buffer you have built in.

### Expected Output

- Completed Part 1 table with all five error budget calculations
- Written answers to all questions in Part 2 scenarios A, B, and C
- Your SLI/SLO/SLA design from Part 3

### Common Mistakes

**Setting SLOs at 100%:** There is no error budget at 100%. Planned maintenance, deployments, and inevitable transient failures make 100% unachievable and unmaintainable as a target. Start at 99.9% and adjust based on measured baseline.

**Confusing SLO and SLA:** The SLO is internal and lower-stakes to breach (it triggers a reliability review). The SLA is external and higher-stakes to breach (it triggers contractual penalties). They are not the same number.

**Ignoring the measurement window:** An SLO of 99.9% over a 30-day window is very different from 99.9% over a 1-hour window. The time window determines how much a single incident affects your budget.

## Lab 1.C: IaC Drift Detection Exercise

### Objective

Understand configuration drift by creating it deliberately, detecting it, and then eliminating it using Terraform. This exercise makes abstract IaC concepts concrete.

### What You Need

- An AWS Free Tier account (required)
- AWS CLI configured with your credentials (`aws configure`)
- Terraform installed (version 1.5 or later)
- A terminal

**Installation references:**
- AWS CLI: `https://docs.aws.amazon.com/cli/latest/userguide/install-cliv2.html`
- Terraform: `https://developer.hashicorp.com/terraform/downloads`

### Prerequisites

- Completed Lab F.A from the Foundation Module
- Basic familiarity with the terminal
- AWS Free Tier account set up (from the Required Tools list in the course overview)

**Cost note:** This lab uses only AWS Free Tier resources. An EC2 t2.micro instance and a security group. Destroy all resources at the end of the lab to avoid charges.

---

### Step 1: Create Your Working Directory

```bash
mkdir ~/devops-lab-iac
cd ~/devops-lab-iac
```

### Step 2: Write the Initial Terraform Configuration

Create a file called `main.tf`:

```bash
cat > main.tf << 'EOF'
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

resource "aws_security_group" "lab_sg" {
  name        = "devops-lab-sg"
  description = "Security group for IaC drift lab"

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "SSH access"
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name        = "devops-lab-sg"
    Environment = "lab"
    ManagedBy   = "terraform"
  }
}

output "security_group_id" {
  value = aws_security_group.lab_sg.id
}
EOF
```

### Step 3: Initialize and Apply

```bash
# Initialize Terraform and download the AWS provider
terraform init

# Preview what Terraform will create
terraform plan

# Review the plan output carefully before proceeding
# You should see: Plan: 1 to add, 0 to change, 0 to destroy.

# Apply the configuration
terraform apply
# Type 'yes' when prompted
```

**Expected output:**

```
Apply complete! Resources: 1 added, 0 changed, 0 destroyed.

Outputs:
security_group_id = "sg-0abc123def456789"
```

Note the security group ID in your output. You will need it in the next step.

### Step 4: Verify the Resource in AWS

```bash
# List security groups and find yours
aws ec2 describe-security-groups \
  --filters "Name=tag:Name,Values=devops-lab-sg" \
  --query "SecurityGroups[*].{ID:GroupId,Name:GroupName,Description:Description}" \
  --output table
```

You should see your security group. Currently it has one inbound rule (SSH on port 22).

### Step 5: Introduce Configuration Drift Manually

This simulates what happens when an engineer makes a manual change directly in AWS without updating the Terraform code. It happens constantly in real environments.

```bash
# Get your security group ID from the Terraform output
SG_ID=$(terraform output -raw security_group_id)

echo "Security group ID: $SG_ID"

# Add an HTTP rule directly via AWS CLI, bypassing Terraform
aws ec2 authorize-security-group-ingress \
  --group-id "$SG_ID" \
  --protocol tcp \
  --port 80 \
  --cidr "0.0.0.0/0"

echo "Manual rule added. Drift has been introduced."
```

### Step 6: Detect the Drift

Now run `terraform plan` again. Terraform compares its state file (which shows only the SSH rule) against what actually exists in AWS (SSH rule plus the manually added HTTP rule).

```bash
terraform plan
```

**Expected output (read this carefully):**

```
aws_security_group.lab_sg: Refreshing state...

Terraform used the selected providers to generate the following execution plan.

  ~ update in-place

  ~ resource "aws_security_group" "lab_sg" {
        id = "sg-0abc123def456789"
      ~ ingress = [
          - {
              - cidr_blocks = ["0.0.0.0/0"]
              - from_port   = 80
              - protocol    = "tcp"
              - to_port     = 80
            },
            {
              + cidr_blocks = ["0.0.0.0/0"]
              + from_port   = 22
              + protocol    = "tcp"
              + to_port     = 22
            },
        ]
    }

Plan: 0 to add, 1 to change, 0 to destroy.
```

Terraform has detected the drift. The `-` sign shows what Terraform will remove (the manually added HTTP rule that is not in the code). The `+` sign shows what Terraform will keep (the SSH rule that is declared in code).

**This is the key lesson:** Terraform does not consider the manual change authoritative. It considers the code authoritative. On the next `apply`, the manual change will be removed.

### Step 7: Observe What `terraform apply` Would Do

Run `terraform apply` to enforce the declared state and eliminate the drift:

```bash
terraform apply
# Type 'yes' when prompted
```

Terraform removes the manually added HTTP rule. The security group now matches the code exactly.

Verify:

```bash
aws ec2 describe-security-groups \
  --filters "Name=tag:Name,Values=devops-lab-sg" \
  --query "SecurityGroups[*].IpPermissions" \
  --output json
```

You should see only the SSH rule (port 22). The HTTP rule is gone.

### Step 8: Fix the Drift the Right Way

Now add the HTTP rule the correct way: through the Terraform code.

Edit `main.tf` and add a second ingress block inside the `aws_security_group` resource:

```hcl
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTP access"
  }
```

Apply the change:

```bash
terraform plan
# You should see: Plan: 0 to add, 1 to change, 0 to destroy.
# The change is adding the HTTP ingress rule through code.

terraform apply
```

The HTTP rule now exists in both AWS and in the Terraform code. There is no drift.

### Step 9: Clean Up

Always destroy lab resources when you are done.

```bash
terraform destroy
# Type 'yes' when prompted
```

Verify the security group is gone:

```bash
aws ec2 describe-security-groups \
  --filters "Name=tag:Name,Values=devops-lab-sg" \
  --query "SecurityGroups[*].GroupId" \
  --output text

# Expected output: (empty)
```

### Common Issues and Troubleshooting

**`terraform init` fails with provider download error:**
Check your internet connection. If you are behind a corporate proxy, set `HTTPS_PROXY` environment variable. Alternatively, pre-download the provider: `terraform providers mirror ./providers` and configure a filesystem mirror.

**`aws configure` credentials not working:**
Run `aws sts get-caller-identity` to verify your credentials are valid. If this returns an error, your AWS access key or secret key is incorrect. Re-run `aws configure` with the correct values from your AWS IAM console.

**Security group already exists error:**
If you ran this lab before and did not destroy, the `devops-lab-sg` security group may still exist. Run `terraform import aws_security_group.lab_sg <sg-id>` to import it into state, then run `terraform destroy`.

**`terraform apply` says "No changes" after introducing drift:**
Terraform's state file may be stale. Run `terraform refresh` to force Terraform to re-read the current AWS state, then run `terraform plan` again.

## Lab 1.D: DevOps Lifecycle Mapping for a Real Service

### Objective

Take a concrete application scenario and map every stage of the DevOps lifecycle to specific tools, automation points, responsible parties, and potential bottlenecks. This is the kind of analysis you will be asked to do when joining a new team or designing a new delivery pipeline.

### What You Need

- Paper, a whiteboard, or draw.io (free at app.diagrams.net)
- Approximately 75 minutes
- No terminal required

### Prerequisites

Read the entire Section 1.1 study note before starting. This lab synthesizes all six topics.

---

### The Scenario

You are joining a small engineering team (six engineers, one DevOps engineer, one QA engineer) at a company building a SaaS project management tool. The application is a Python Flask backend with a React frontend. It runs on AWS. The current state of their delivery process:

- Code is hosted on GitHub
- Deployments happen manually: an engineer SSH-es into the EC2 instance and runs `git pull && sudo systemctl restart app`
- There are no automated tests running in CI
- There is one environment: production (no staging)
- Monitoring consists of checking the server manually when a customer reports a problem
- Incidents take between 4 and 8 hours to resolve because the team finds out about them from customer emails

This is a real situation. Many teams at small companies operate exactly this way.

A strong answer to this lab is specific, not generic. Saying "add monitoring" is weak. The goal is to practice thinking like a DevOps engineer: every improvement is concrete, has an owner, and has a measurable success condition.
