## 1. Software Development Lifecycle (SDLC)

### What is SDLC?

The Software Development Lifecycle (SDLC) is a structured framework that guides how software is planned, built, tested, deployed, and maintained. It gives teams a repeatable process so the final product actually meets user requirements, stays within budget, and ships without critical defects.

Think of it like a recipe: you follow defined steps in order, and the output is predictable. Skip steps or change the order, and things break.


**Why SDLC matters?**

1. Breaks a complex project into manageable phases with clear ownership and timelines.
2. Gives every team member, developers, testers, ops engineers, and clients, a shared understanding of where the project stands.
3. Catches problems early, when they are cheap to fix.
4. Creates an auditable paper trail for regulated industries.
5. Reduces rework by aligning on requirements before code is written.

---

### The Seven Phases of SDLC

**Phase 1: Requirement Collection**

The team gathers and documents what the software must do. This is the most critical phase — wrong requirements here cascade into every phase that follows.

**Responsible teams:** Product Managers, Business Analysts, and Stakeholders.

**Tools:** Jira (task and user story tracking), Confluence (requirement documentation and meeting notes).

**Example:** For a fitness tracking app, requirements might include step counting, heart rate monitoring, and syncing with wearable devices over Bluetooth.


**Phase 2: Feasibility Study**

The team evaluates whether the project is technically and financially viable before committing resources.

**Factors assessed:** development cost, available technical expertise, legal and regulatory constraints, and timeline realism.

**Example:** A startup building a smart home app must confirm it can afford the hardware integrations, comply with IoT security regulations, and hire engineers with the right embedded systems skills.


**Phase 3: System Design**

The team translates requirements into a technical blueprint: system architecture, database schema, UI/UX wireframes, API specifications, and infrastructure design.

**Responsible teams:** System Architects, UI/UX Designers, Database Designers.

**Example:** For a ride sharing app, architects choose a microservices architecture (User Service, Trip Service, Pricing Service), design the relational schema, and produce wireframes for the booking flow.


**Phase 4: Coding (Implementation)**

Developers write the actual application code based on the approved design.

**Responsible teams:** Frontend Developers, Backend Developers.

**Example:** Backend developers write REST APIs in Python/Django. Frontend developers build the interface in React. Database queries run against PostgreSQL.


**Phase 5: Testing**

The QA team verifies that the software works correctly, handles edge cases, performs under load, and meets the original requirements.

**Responsible teams:** QA Engineers, Test Automation Engineers.

**Types of testing:**
- Unit Testing: Tests individual functions in isolation.
- Integration Testing: Tests how modules work together (e.g., does the shopping cart correctly communicate with the payment gateway?).
- User Acceptance Testing (UAT): Real users or stakeholders validate the product before release.
- Performance/Load Testing: Validates behavior under high traffic.

**Tools:** Selenium (browser automation), Postman (API testing), JUnit/PyTest (unit testing), k6/JMeter (load testing).


**Phase 6: Deployment**

The tested application is released to the production environment, where real users can access it.

**Responsible teams:** DevOps Engineers, System Administrators.

**Example:** A CI/CD pipeline (GitHub Actions or Jenkins) automates building, testing, and deploying the application to AWS. The deployment runs without manual steps.


**Phase 7: Maintenance**

After deployment, the team keeps the software running, secure, and aligned with evolving user needs.

**Responsible teams:** Support Engineers, Maintenance team.

**Types of maintenance:**
- Corrective: Fix bugs (e.g., resolving a crash on iOS 17).
- Adaptive: Update compatibility (e.g., support for Android 15).
- Perfective: Add improvements based on user feedback (e.g., adding dark mode).

---

### SDLC and DevOps

DevOps does not replace SDLC; it accelerates it. Traditional SDLC treats phases as sequential handoffs between siloed teams. DevOps collapses those handoffs through automation, shared ownership, and continuous feedback. The same seven phases still happen; they just happen faster, in smaller batches, and with far less manual coordination.


| SDLC Phase | DevOps Practice That Accelerates It |
|---|---|
| Requirement Collection | User stories tracked in Jira; linked to Git commits |
| Design | Architecture documented as code (diagrams-as-code) |
| Coding | Feature branches; pull requests with automated checks |
| Testing | Automated unit, integration, and security tests in CI |
| Deployment | CI/CD pipeline deploys on every merge to main |
| Maintenance | Monitoring alerts, detecting, and routing issues automatically |

## 2. Development Models

---
### I. Waterfall Model

Waterfall is a linear, sequential model. Each phase must be completed and signed off on before the next begins. There is no going back without a formal change request.


**Characteristics:**
- Phases flow strictly top down: Requirements > Design > Coding > Testing > Deployment > Maintenance.
- Heavy upfront documentation; final product is delivered at the end.
- Changes mid project trigger a formal Change Control Request (CCR) and are expensive to implement.


**When Waterfall makes sense:**
- Requirements are fixed, well understood, and unlikely to change (e.g., government contracts, regulatory compliance systems, embedded firmware).
- Heavy documentation is a legal or contractual requirement.
- Budget and timeline are locked before development starts.


**Example: E-commerce site built with Waterfall:**

| Phase | Duration | Output |
|---|---|---|
| Requirements | 3 weeks | Approved Software Requirements Specification (SRS), stakeholder sign off |
| System Design | 4 weeks | Architecture document, DB schema, UI mockups, API spec |
| Coding | 10 weeks | Working codebase in dev environment, unit test reports |
| Testing | 4 weeks | Test reports, signed-off release candidate |
| Deployment + Maintenance | 1 week + ongoing | Live site, runbooks, support logs |


**Challenges:**
- Inflexibility: if the market changes or a requirement is wrong, pivoting is costly.
- Late discovery: usability problems and missing features only surface during testing or after launch.
- High rework cost: a change in requirements can force multiple completed phases to be revisited.

---

### II. Agile Model

Agile is an iterative, incremental model. Instead of delivering everything at the end, the team ships working software in short cycles called sprints (typically one to two weeks). Requirements evolve based on feedback; the plan adapts continuously.


**Characteristics:**
- Work is broken into sprints. Each sprint produces a working, potentially shippable increment.
- Stakeholders review progress at the end of every sprint and provide feedback that shapes the next one.
- Emphasis on cross functional collaboration: developers, testers, and operations work together throughout.
- Requirements are captured as user stories: "As a [user], I want [feature] so that [benefit]."


**Core Agile ceremonies:**
- Sprint Planning: Team selects stories from the backlog and commits to completing them in the sprint.
- Daily Stand up: 15 minute sync. Each person answers: what did I do yesterday, what will I do today, what is blocking me?
- Sprint Review: Team demos the working increment to stakeholders and collects feedback.
- Sprint Retrospective: Team reflects on the process and agrees on one or two concrete improvements.


**When Agile makes sense:**
- Requirements are expected to change or are not fully understood upfront (most modern software).
- Fast time to market matters.
- Continuous stakeholder engagement is possible.


**Example: E-commerce site built with Agile:**

| Sprint | Stories Delivered | Feedback Collected |
|---|---|---|
| 1 | User login/logout, basic product catalog | "Add social login (Google, Facebook)" |
| 2 | Shopping cart, product search and filtering | "Support coupon codes at checkout" |
| 3 | Checkout flow, payment gateway, order history | "Show confirmation email link on success screen" |

Each sprint ends with a demo. Feedback goes directly into the backlog and can be prioritized for the next sprint; no formal change request is required.


**Challenges:**
- Requires strong team discipline to avoid scope creep.
- Demands continuous stakeholder availability and engagement.
- Needs robust test automation (CI/CD) to maintain quality across many small releases.

---

### Waterfall vs. Agile: When to Choose Which

| Factor | Choose Waterfall | Choose Agile |
|---|---|---|
| Requirements | Fixed and fully understood | Evolving or partially known |
| Feedback loops | Periodic, formal reviews | Continuous, every sprint |
| Change tolerance | Low; changes are expensive | High; changes are expected |
| Documentation | Heavy, required for compliance | Lightweight, just enough |
| Delivery | Single delivery at project end | Incremental delivery every sprint |
| Best for | Compliance systems, fixed contracts | SaaS, mobile apps, most modern software |

## 3. DevOps

### What Is DevOps?

DevOps is a cultural philosophy and a set of engineering practices that brings software development (Dev) and IT operations (Ops) together. The goal is to deliver software faster, more reliably, and with fewer failures.

Before DevOps, development and operations were separate silos:
- Developers wrote code and "threw it over the wall" to operations.
- Operations deployed it manually, often weeks or months after development finished.
- When things broke in production, blame flowed between teams.

DevOps eliminates those handoffs through shared ownership, automation, and continuous feedback.


**The eight dimensions of DevOps:**

1. **Culture and Collaboration:** shared goals, psychological safety, no blame culture.
2. **Automation:** build, test, provision, deploy, and remediate automatically.
3. **Continuous Integration and Continuous Delivery (CI/CD):** code integrates and ships frequently in small, safe batches.
4. **Observability and Feedback:** metrics, logs, and traces reveal system behavior in real time.
5. **Lean Flow:** Small batch sizes and fast feedback loops reduce waste.
6. **Reliability Engineering:** Service Level Objective (SLOs) and error budgets govern the balance between velocity and stability.
7. **Security Shift Left (DevSecOps):** security checks run in development and CI, not just before release.
8. **Continuous Learning:** blameless postmortems and experimentation drive improvement.


**Silo mentality: The problem DevOps solves:**

| Aspect | Description |
|---|---|
| Causes | Misaligned incentives, legacy org structures, unclear ownership, blame culture |
| Symptoms | Slow handoffs, manual deployments, inconsistent environments, delayed incident resolution |
| Impacts | Long lead times, high change failure rate, low morale, fragile releases |
| DevOps remedies | Cross functional teams, shared metrics (DORA), blameless postmortems, self service platforms |

---

### The DevOps Lifecycle

The DevOps lifecycle is often shown as an infinity loop to represent continuous, never ending delivery. It covers eight stages:

`Plan > Code > Build > Test > Release > Deploy > Operate > Monitor > (back to Plan)`

Each stage has dedicated tools. As a DevOps engineer, you will work across all of them.


**Stage 1: Plan**

Teams define requirements, prioritize features, and track work. Small, well-defined tasks reduce confusion and enable fast delivery.

**Tools:** Jira, GitHub Issues, Confluence.

**Practice:** Every piece of work ties to a user story. Commit messages reference ticket numbers so changes are traceable from requirement to production.


**Stage 2: Code (Continuous Development)**

Developers write code on short-lived feature branches and commit frequently. Small commits are easier to review, test, and roll back.

**Tools:** Git, GitHub/GitLab, VS Code, IntelliJ.

**Practice:** Trunk based development (committing to main frequently with feature flags) or short lived feature branches merged via pull request. Avoid long running branches: they cause painful merge conflicts.

- Short lived feature branch workflow
```bash
git checkout -b feat/payment-retry                        # create and switch to a new feature branch
git commit -m "feat: add idempotent payment retry logic"  # commit staged changes with a descriptive message
git push origin feat/payment-retry                        # push the branch to remote
```


**Stage 3: Build (Continuous Integration)**

On every push or pull request, a CI pipeline runs automatically: compile the code, run unit tests, run static analysis, scan for known vulnerabilities, and produce a build artifact (e.g., a container image or a JAR file).

**Tools:** Jenkins, GitHub Actions, GitLab CI.

CI pipeline flow:
```
push to branch
  > checkout code
  > install dependencies
  > compile / build
  > unit tests
  > static analysis (lint, SAST)
  > dependency vulnerability scan (SCA)
  > build container image
  > push to registry
```

If any step fails, the pipeline stops and the developer is notified immediately. This prevents broken code from progressing.


**Stage 4: Test (Continuous Testing)**

Automated quality gates run at multiple levels. Tests are not a phase that happens at the end; they run continuously throughout development.

| Test Type | What It Checks | Tools |
|---|---|---|
| Unit | Individual functions in isolation | JUnit, PyTest |
| Integration | How modules work together | TestNG, PyTest |
| End to End (E2E) | Full user flows in a browser | Selenium, Playwright |
| Performance | Behavior under load | k6, JMeter |
| Security (SAST) | Code level vulnerabilities | Semgrep, SonarQube |
| Container scanning | Known CVEs in images | Trivy, Grype |
| Policy | Infrastructure compliance | Checkov, OPA |

Shift left testing means running tests earlier in the pipeline rather than waiting until a dedicated test phase. A security issue found in CI costs far less to fix than one found after deployment.


**Stage 5: Release and Deploy (Continuous Delivery / Deployment)**

- Continuous Delivery: Every passing build produces a release candidate that could be deployed to production. A human approves the final promotion.
  
- Continuous Deployment: every passing build deploys to production automatically, with no human approval step.

Deployment strategies:

| Strategy | How It Works | Risk Level |
|---|---|---|
| Rolling | Replace old instances one at a time | Low |
| Blue/Green | Run two identical environments; switch traffic instantly | Very Low |
| Canary | Route a small percentage of traffic to the new version first | Very Low |
| Feature Flag | Deploy code to production but hide the feature behind a toggle | Very Low |

**Tools:** Jenkins, GitHub Actions, ArgoCD, Helm, Terraform, and Ansible.


**Stage 6: Operate**

Once deployed, the application must be kept running. This includes configuration management, patching, scaling, and incident response.

**Tools:** Ansible (configuration), Terraform (infrastructure), Kubernetes (container orchestration).

**Practice:** Infrastructure is defined as code; no manual changes are made directly to servers. Any change goes through version control, review, and the pipeline.


**Stage 7: Monitor (Continuous Monitoring and Feedback)**

Observability tools collect metrics, logs, and traces from the running system. Dashboards and alerts give the team visibility into availability, performance, and errors.

**The three pillars of observability:**
- Metrics: numeric time series data (CPU usage, request rate, error count). Answer: "What is the system doing right now?"
  
- Logs: timestamped records of discrete events. Answer: "What happened at 14:32 UTC?"

- Traces: the path a request took through distributed services. Answer: "Where did this request spend its time?"

**Tools:** Prometheus and Grafana (metrics), Elastic Stack (logs), OpenTelemetry + Jaeger (traces).

**Practice:** Define Service Level Indicators (SLIs) and Service Level Objectives (SLOs) before the system goes live. An SLI is a measurement (e.g., 95th percentile latency). An SLO is the target (e.g., p95 latency must stay below 300 ms). When an SLO is breached, alerts fire, and the on call engineer responds.


**Stage 8: Learn and Improve**

Monitoring data, incident reports, and sprint retrospectives feed back into planning. The loop is continuous; there is no final state.

**Practice:** Blameless postmortems after incidents. The goal is to find process and system failures, not to assign blame to individuals. Every incident produces action items: runbook improvements, new automated tests, alerting changes, or infrastructure fixes.

## 4. Infrastructure as Code (IaC)

### What Is IaC?

Infrastructure as Code is the practice of managing and provisioning infrastructure such as servers, networks, databases, and load balancers, using machine readable configuration files rather than manual processes or interactive tools.

Before IaC, a system administrator would log into a server and run commands by hand to configure it. That process was slow, error-prone, undocumented, and impossible to reproduce exactly. IaC solves all of these problems.


### How IaC Works?

There are two approaches:


**Declarative (Desired State):** You describe what you want the infrastructure to look like. The tool figures out how to make it happen and reconciles the current state with your declared state.

Tools: Terraform, AWS CloudFormation, Pulumi, Puppet.

- Terraform example: declare an EC2 instance
```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c02fb55956c7d316"
  instance_type = "t2.micro"
}
```

Run `terraform apply` and Terraform creates the instance. Run it again, and nothing changes; the instance already matches the declaration. This property is called idempotency.


**Imperative (Step by Step):** You write the exact commands to execute in sequence. The tool runs them in order.

Tools: Ansible (primarily), Bash scripts.

- Ansible example: install and start nginx
```yaml
- name: Install nginx
  dnf:
    name: nginx
    state: present

- name: Start nginx
  service:
    name: nginx
    state: started
    enabled: true
```

In practice, Ansible is used alongside Terraform: Terraform provisions the infrastructure, Ansible configures it.


### Benefits of IaC


**Speed:** Provision a complete environment (servers, network, database, load balancer) in minutes, not days. Spin up identical environments on demand for development, testing, staging, and production.


**Consistency:** Every environment is created from the same code. Configuration drift, where servers gradually differ from each other due to manual changes, is eliminated.


**Auditability:** IaC files live in Git. Every change has a commit message, an author, a timestamp, and a diff showing exactly what changed. Rollback is a `git revert`.


**Reproducibility:** A new team member can clone the repository and recreate the entire infrastructure with one command. No institutional knowledge locked in someone's head.


**Cost reduction:** Environments can be torn down when not in use and recreated on demand. No idle infrastructure pays for itself overnight.


### IaC Tools

| Category | Tools | Primary Use |
|---|---|---|
| Multi cloud provisioning | Terraform, Pulumi | Create and manage cloud resources across any provider |
| Cloud native provisioning | AWS CloudFormation, Azure ARM, GCP Deployment Manager | Provision resources within a single cloud provider |
| Configuration management | Ansible, Puppet, Chef, Salt | Configure software on existing servers |

## 5. Microservices Architecture

### What Is a Microservice?

A microservice is an independently deployable service that handles one specific business capability. A microservices architecture is an application built as a collection of these services, each running in its own process and communicating over well defined APIs.

This is the opposite of a monolithic architecture, where all functionality lives in a single codebase and is deployed as one unit.

---

### Monolith vs. Microservices

| Aspect | Monolithic | Microservices |
|---|---|---|
| Codebase | Single repository, single deployment unit | Multiple repositories, independent deployments |
| Scaling | Scale the entire application | Scale individual services based on demand |
| Failure impact | One bug can take down everything | Failures are isolated to one service |
| Technology | One stack for the whole application | Each service can use the best tool for its job |
| Deployment speed | Slow the whole app must be built and deployed | Fast deploy one service without touching others |
| Team structure | All developers work in the same codebase | Small teams own specific services |
| Complexity | Simple to start; hard to scale | More complex infrastructure; scales well |

---

### Why Microservices Matter for DevOps?

Microservices are the dominant architecture in modern cloud native systems. As a DevOps engineer, you will build pipelines, containers, and deployment configurations for microservices environments.

**Major implications:**
- Each service gets its own CI/CD pipeline.
- Each service runs in its own container and is independently deployable.
- Services communicate over HTTP REST or gRPC, or asynchronously through a message broker (Kafka, RabbitMQ).
- Observability becomes critical: when a request fails, you need distributed tracing to identify which service is responsible.

---

### Example: Ride Sharing Platform

A ride sharing app built with microservices might look like this:

| Service | Responsibility | Data Store |
|---|---|---|
| User Service | Manages rider and driver profiles and authentication | PostgreSQL |
| Trip Service | Matches riders to drivers, calculates ETA, stores trip records | PostgreSQL + Redis (cache) |
| Pricing Service | Computes dynamic fares based on demand, distance, surge | Redis |
| Notification Service | Sends ride confirmed events to email/SMS workers via Kafka | Kafka |
| API Gateway | Routes all external client requests to the correct service | N/A |

---

**Major design decisions:**

- Each service owns its own data store. No service directly queries another service's database.

- Services communicate synchronously (gRPC for low latency calls like authentication) or asynchronously (Kafka for events that don't need an immediate response).

- Each service runs in Docker containers managed by Kubernetes, enabling independent scaling (e.g., spin up more Trip Service pods during rush hour).

---

**Resilience patterns used in production microservices:**

- Circuit Breaker: if the Pricing Service is down, the Trip Service falls back to cached pricing rather than failing.

- Retry with Exponential Backoff: transient network failures retry automatically with increasing delays.

- Bulkhead: CPU intensive operations run on dedicated nodes so they cannot starve other services of resources.


### Microservices and Observability

In a monolith, a failed request shows up in one log file. In microservices, a single user request might touch eight different services. Finding where it failed requires distributed tracing: each service tags its logs and spans with a shared trace ID, so you can follow the full request path across all services.

**Tools you will use:** Prometheus and Grafana (metrics per service), the Elastic Stack (centralized logs), OpenTelemetry (standardized tracing instrumentation).

## 6. SRE vs. DevOps

### What Is Site Reliability Engineering (SRE)?

Site Reliability Engineering (SRE) was created at Google around 2003 to solve a specific problem: how do you operate large scale, highly reliable services without an ever growing manual operations team?

SRE's answer is to treat operations as a software engineering problem. SREs write code to automate operational tasks, define quantitative reliability targets, and use those targets to make data driven decisions about when to release new features and when to focus on stability.

Google's SRE book states it directly: "If DevOps is a philosophy, SRE is a prescriptive way of accomplishing that philosophy."

---

### Core SRE Concepts

**Service Level Indicator (SLI):** A specific, measurable metric that represents one aspect of service quality. Examples: request success rate, 95th percentile latency, availability percentage.


**Service Level Objective (SLO):** A target value for an SLI. Example: 99.9% of requests succeed (meaning no more than 0.1% may fail over a rolling 30 day window). SLOs are internal targets, agreed between the SRE team and product teams.


**Error Budget:** The amount of unreliability an SLO allows. If the SLO is 99.9% availability, the error budget is 0.1%, roughly 43 minutes of downtime per month. When the error budget is exhausted, the team stops deploying new features and focuses exclusively on reliability improvements. This gives both development and operations teams a shared, objective measure to negotiate around.


**Toil:** Repetitive, manual, automatable work that does not produce lasting value. SREs aim to keep toil below 50% of their working time. Everything above that threshold gets automated.


**Blameless Postmortem:** A structured analysis of an incident that focuses on system and process failures, not individual mistakes. Every significant incident produces a written postmortem with a timeline, root cause analysis, and concrete action items.

---

### DevOps vs. SRE

| Aspect | DevOps | SRE |
|---|---|---|
| Nature | Cultural philosophy and set of practices | A specific engineering role and implementation model |
| Focus | Collaboration, automation, continuous delivery | Reliability, scalability, eliminating operational toil |
| Key concepts | CI/CD, IaC, shared ownership, feedback loops | SLOs, error budgets, toil reduction, blameless postmortems |
| Metrics | Deployment frequency, lead time, change failure rate | Availability, latency, error rate, MTTR |
| Philosophy | "You build it, you help run it" | "Operations is a software problem" |
| Prescriptiveness | Principles and practices; teams implement them as they choose | Specific job descriptions, 50% toil cap, SLO framework |
| Origin | Community-driven; popularized by the 2009 Velocity conference talk by Allspaw and Hammond | Created at Google in 2003; documented in the Google SRE book (2016) |

---

**The Major distinction:**

DevOps is the philosophy that Dev and Ops should collaborate closely. SRE is one concrete, opinionated implementation of how to do that in practice. Many organizations practice DevOps broadly while having dedicated SRE teams focused specifically on reliability.

---

### SRE in Practice: CDN Outage Incident

This example illustrates how SRE concepts apply during a real incident.

**Situation:** A Content Delivery Network (CDN) provider deployed a new Web Application Firewall (WAF) rule. The rule caused CPU spin loops on packet inspection across half the edge servers, producing widespread 502 errors for end users.

**Timeline:**

13:42 UTC: Monitoring detects 502 error rates exceeding 5%, breaching the SLO. PagerDuty alerts fire and page the on call SRE.

13:50 UTC: The on call SRE declares a Major Incident and assembles an incident response team (Networking, Security, and Operations engineers).

14:05 UTC: Initial hypothesis (DDoS attack) is ruled out. Log analysis reveals the new WAF rule is causing a CPU spin loop on packet inspection.

14:35 UTC: Root cause confirmed. The faulty WAF rule is rolled back globally. Servers recover within 10 minutes. Error rates return within SLO.

14:50 UTC: Incident resolved. Postmortem begins.


**Postmortem action items:**

- Implement canary rollout for all firewall rule changes (deploy to 1% of edge servers first, monitor for 30 minutes, then promote).

- Add synthetic traffic tests against the WAF in staging, running every 5 minutes.

- Manage all firewall configurations in Git with pull request reviews (IaC).

- Update the on call runbook with a "verify CPU load on edge nodes before global rollout" step.

- Automate rollback: if CPU load or error rates spike above threshold within 10 minutes of a WAF change, revert automatically.

This postmortem is blameless. The action items target the system: the absence of canary deployment for WAF rules, the lack of synthetic monitoring, and the missing runbook step. No individual is blamed for the outage.
