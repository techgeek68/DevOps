## Theory

## 1. Virtualization

Virtualization is the process of creating software based versions of physical resources such as servers, storage, and networks. It allows multiple operating systems and applications to run on a single physical machine, making better use of hardware that would otherwise sit idle.

Before virtualization, a physical server typically ran one operating system and one workload. The hardware was underutilized, expensive to maintain, and difficult to reproduce. Virtualization solved all three problems at once.

---

### Why Virtualization Matters for DevOps?

| Benefit | Practical Meaning |
|---|---|
| Resource efficiency | Run multiple VMs on one physical machine instead of one workload per server |
| Isolation | A fault, crash, or misconfiguration in one VM does not affect others |
| Environment consistency | Recreate identical dev, test, staging, and production environments from the same base image |
| Faster provisioning | Deploy a pre configured VM in minutes instead of hours of manual setup |
| Scalability | Spin VMs up or down to match workload demand without touching hardware |
| Cost reduction | Fewer physical servers, lower power consumption, less data center space |
| Disaster recovery | VM snapshots capture state at a point in time; roll back to a clean state instantly if something breaks |

In a DevOps context, virtualization is the foundation for lab environments, CI/CD build agents, Kubernetes nodes, and local development machines that mirror production.

---

## 2. Hypervisors

A hypervisor (also called a Virtual Machine Monitor) is the software layer that creates and manages virtual machines. It sits between the physical hardware and the guest operating systems, allocating CPU, RAM, storage, and network resources to each VM and keeping them isolated from one another.

### Type 1: Bare Metal Hypervisor

A Type 1 hypervisor runs directly on the physical hardware. There is no underlying host operating system. The hypervisor itself is the operating system, and VMs run on top of it.

Type 1 hypervisors offer higher performance, stronger isolation, and better security because no host OS layer can be compromised or consume resources.

Type 1 hypervisors are used in production data centers, cloud provider infrastructure, and enterprise server virtualization.

Examples: **VMware ESXi**, **Microsoft Hyper-V**, **Xen**, **KVM** (Kernel based Virtual Machine), **Proxmox**

**KVM:** 

KVM is a Linux kernel module, integrated into the kernel since version 2.6.20. When KVM is loaded, the Linux kernel itself becomes a Type 1 hypervisor with direct access to hardware virtualization extensions (Intel VT-x, AMD-V). QEMU is a separate user space tool that works alongside KVM to handle device emulation. The combination is referred to as KVM/QEMU. KVM alone is the hypervisor; QEMU provides the hardware interface that makes it usable.

**Hyper V:**

Microsoft Hyper-V is a Type 1 hypervisor. When Hyper-V is enabled on a Windows host, the hypervisor inserts itself between the hardware and the OS. The Windows installation that you see becomes a privileged management partition, not a standard host OS. This is confirmed in the official Microsoft documentation: "Hyper-V features a Type 1 hypervisor based architecture."

---

### Type 2: Hosted Hypervisor

A Type 2 hypervisor runs as an application on top of an existing host operating system. The host OS manages hardware resources, and the hypervisor sits on top of it.

Type 2 hypervisors are easier to install and use, making them well suited for local development, student lab environments, and desktop virtualization. They add overhead compared to Type 1 because all hardware access passes through the host OS.

Examples: **VMware Workstation Pro**, **VirtualBox**, **Parallels Desktop**, **UTM**

### Comparison: Type 1 vs Type 2

| Feature | Type 1 (Bare Metal) | Type 2 (Hosted) |
|---|---|---|
| Runs on | Physical hardware directly | Host operating system |
| Performance | High; minimal overhead | Moderate; host OS adds overhead |
| Security | Stronger isolation | Dependent on host OS security |
| Setup complexity | Higher; dedicated hardware needed | Low; installs like any application |
| Use case | Production servers, cloud infrastructure | Developer labs, learning environments |
| Examples | VMware ESXi, Hyper V, KVM | VMware Workstation Pro, VirtualBox |

---

### Comparison: VMs vs. Containers

Both VMs and containers provide isolation, but they work differently and serve different purposes.

| Aspect | Virtual Machine | Container |
|---|---|---|
| Isolation unit | Full OS + kernel | Process namespace on shared host kernel |
| Size | Gigabytes | Megabytes |
| Startup time | Minutes | Seconds |
| OS | Each VM has its own OS | All containers share the host kernel |
| Security isolation | Strong hardware level isolation | Process level isolation |
| Use case | Full system environments, multiple OS types | Microservices, CI/CD, packaging applications |

VMs and containers are complementary. In this course, you run containers (Module 5) inside VMs (this module). In production, many organizations run containers inside VMs on cloud infrastructure.

---

## Comparison: WSL vs VM

| Feature | WSL2 | VM |
|---|---|---|
| **Type** | Lightweight Hyper V VM with real Linux kernel | Full hardware emulation |
| **Kernel** | Microsoft maintained Linux kernel fork | Full guest OS kernel |
| **Performance** | Near native | Hypervisor overhead |
| **Boot time** | Seconds | Minutes |
| **Resource usage** | Lightweight | Heavy (dedicated RAM/CPU) |
| **GUI support** | Good (via WSLg) | Full desktop support |
| **Isolation** | Partial (Hyper-V boundary) | Full |
| **Networking** | vEth + NAT (mirrored mode available) | Separate network stack |
| **File system** | Own ext4 disk + `/mnt/c` Windows access | Separate disk image |
| **Snapshot/rollback** | N/A | Available |
| **Hardware access** | Limited | Depends on hypervisor |
| **Use case** | Dev tools, CLI, scripting | Full OS testing, labs, servers |

> WSL2 is a real Linux kernel in a lightweight VM, optimized for developer workflows. VMs offer full isolation and hardware control, better suited for infrastructure and lab environments.

## Part 1: Check Your Hardware Before You Begin
 
### Verify CPU Virtualization Support

VirtualBox requires **Intel VT-x** or **AMD-V** hardware virtualization enabled in your BIOS/UEFI.

### Windows

Open **Task Manager** (Ctrl+Shift+Esc), navigate to the **Performance** tab, select **CPU**, and confirm **Virtualization: Enabled**.

### Linux

Run the following command to check for flags:

```bash
grep -E "vmx|svm" /proc/cpuinfo | head -1
```
>**Result:** `vmx` (Intel) or `svm` (AMD) confirms support.
>**No output:** Virtualization is disabled or unsupported; check your BIOS.

### macOS (Unix)

Check for Apple Hypervisor Framework support:

```bash
sysctl kern.hv_support
```
>**Result:** `kern.hv_support: 1` confirms it is active.

---

> Troubleshooting: Enabling in BIOS/UEFI

>If disabled, restart your machine, enter the BIOS/UEFI setup (typically **F2, F10, Del, or Esc** at startup), and locate the **Intel Virtualization Technology** or **AMD-V** setting under the **CPU** or **Advanced** tab.

**Install VirtualBox Guest Additions:**
 
Guest Additions give you: automatic display resizing when you resize the VirtualBox window, shared clipboard between host and guest, shared folder support, and better mouse integration. First, install the required build dependencies:
 
```bash
sudo dnf install -y gcc make perl kernel-devel kernel-headers bzip2 elfutils-libelf-devel

```
 
After rebooting, log back in. Then, in the VirtualBox menu bar of the VM window, click **Devices > Insert Guest Additions CD Image**. This mounts the Guest Additions ISO inside the VM.
 
Mount and run the installer:
 
* Create a mount point
```bash
sudo mkdir -p /mnt/vbox
```

* Mount the Guest Additions disc
```bash
sudo mount /dev/cdrom /mnt/vbox
```

* Run the Guest Additions installer
```bash
sudo sh /mnt/vbox/VBoxLinuxAdditions.run
```

* Reboot to activate the new kernel modules
```bash
sudo reboot
```

After rebooting, resize the VirtualBox window. The VM display should automatically resize to match. If it does, Guest Additions are working.

## Full Clone vs. Linked Clone
 
VirtualBox offers two clone types:
 
**Full Clone:** A completely independent copy of the VM. All virtual disk files are duplicated. The clone has no dependency on the base VM. You can delete the base VM and the clone continues to work. Performance is identical to a fresh install. Takes 2 to 5 minutes per clone. Recommended for this course.
 
**Linked Clone:** A small delta file that stores only the differences from a parent snapshot. Provisioning is nearly instant and uses very little disk space. However, the clone depends on the parent VM and its snapshot chain remaining intact. If the parent is deleted or moved, the linked clone breaks.
 
For a persistent lab environment where VMs need to survive independently across weeks or months, use full clones. Linked clones are better suited for short-lived disposable environments.

## Post Clone Configuration
 
Every cloned VM needs a unique hostname and a known IP address. Do this for each clone as you create it.
 
**Step 1: Power on the cloned VM**
 
**Step 2: Change the hostname**
 
```bash
sudo hostnamectl set-hostname jenkins-controller
```
 
Replace `jenkins-controller` with the appropriate name for the VM you are configuring. Verify:
 
```bash
hostnamectl
```
 
The new hostname takes effect for new shell sessions. Log out and back in to see it reflected in the prompt.
 
**Step 3: Assign a static IP on the Host-Only interface**
 
By default, the clone gets a DHCP address on the Host-Only network. This can change after a reboot, making it hard to SSH into the VM consistently. Assign a static IP to prevent this.
 
Use the address plan below to keep things organized. All addresses are in the `192.168.56.0/24` range (the default VirtualBox Host-Only network):
 
| VM Name            | Static IP          |
|--------------------|--------------------|
| DevOps-BaseVM      | 192.168.56.10      |
| control-node       | 192.168.56.11      |
| web-server         | 192.168.56.20      |
| app-server         | 192.168.56.21      |
| jenkins-controller | 192.168.56.30      |
| jenkins-agent      | 192.168.56.31      |
| ansible-node       | 192.168.56.40      |
| container-host     | 192.168.56.50      |
| k8s-control-plane  | 192.168.56.60      |
| k8s-worker-1       | 192.168.56.61      |
| k8s-worker-2       | 192.168.56.62      |
 
To assign a static IP on the Host-Only adapter (typically the second interface, `enp0s8`):
 
```bash
# List all connections to find the Host-Only connection name
nmcli connection show
 
# Modify the connection (replace "Wired connection 2" if the name differs on your system)
sudo nmcli connection modify "Wired connection 2" \
  ipv4.method manual \
  ipv4.addresses 192.168.56.30/24 \
  ipv4.dns "8.8.8.8,8.8.4.4"
 
# Apply the change
sudo nmcli connection down "Wired connection 2"
sudo nmcli connection up "Wired connection 2"
 
# Confirm
ip addr show
```
 
Leave the NAT adapter (enp0s3) on DHCP. It handles outbound internet access and does not need a fixed IP for lab use.
 
**Step 4: Update /etc/hosts on your host machine**
 
Add entries for each VM so you can SSH by name instead of IP. On your host machine (not inside the VM), edit the hosts file:
 
On Linux and macOS: `/etc/hosts`
On Windows: `C:\Windows\System32\drivers\etc\hosts`
 
```
192.168.56.30  jenkins-controller
192.168.56.31  jenkins-agent
192.168.56.60  k8s-control-plane
192.168.56.61  k8s-worker-1
192.168.56.62  k8s-worker-2
```
 
After adding these entries, you can SSH using names:
 
```bash
ssh username@jenkins-controller
```
 
**Step 5: Regenerate the machine ID if needed**
 
VirtualBox generates new MAC addresses for full clones when you select the option in the Clone dialog. For most labs, this is sufficient. For Kubernetes, each node also needs a unique machine ID:
 
```bash
# Check the current machine ID
cat /etc/machine-id
 
# Regenerate
sudo rm /etc/machine-id
sudo systemd-machine-id-setup
cat /etc/machine-id
```

## Managing VMs
 
**Delete a VM permanently:**
 
Right click the VM in VirtualBox Manager. Select **Remove**. When prompted, choose **Delete all files**. This removes all VM files, including snapshots and virtual disks. It cannot be undone.


**Remove from list without deleting files:**
 
Right click and select **Remove**, then choose **Remove only**. The files remain on disk and can be re-added via **Machine > Add**.

 
**Suspend vs. Power Off:**
 
- **Save State** (suspend) writes the current memory to disk and pauses the VM. Resume is fast but consumes disk space equal to the VM's allocated RAM.
  
- **Power Off** cleanly shuts down the guest OS. Slower to start, but uses no extra disk space. For VMs not actively in use, power them off to free RAM on your host.
 
---
 
## Recommendations
 
**Never configure anything course-specific on `DevOps-BaseVM`.** Its only job is to be cloned. If you accidentally install or configure something on it, revert to the `Clean Install` snapshot before creating more clones.
 
**Create VMs as each module requires them,** not all at once. Cloning all 10 VMs up front wastes disk space and creates clutter. A 20 GB VM with a dynamically allocated disk grows over time, and 10 of them can consume hundreds of gigabytes.
 
**Use VirtualBox's Description field to label each VM.** Right click the VM, select **Settings > General > Description**. Record the module it is used for, the hostname, and the assigned IP. This saves time when you come back to the lab after a few days.
